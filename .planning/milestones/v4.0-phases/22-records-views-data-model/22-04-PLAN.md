<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| internet → /api/v1/records/* | First point untrusted HTTP requests hit this domain; `authenticateToken` is the sole gate before any handler runs |
| authenticated user → cross-tenant params (`:id`) | A valid JWT does not imply the referenced collection/record/view belongs to the caller's company — every handler passes `requireCompany(req)` into the service, which re-checks ownership via repository scoping |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-10 | Spoofing / Elevation of Privilege | records.routes.ts (all 13 routes) | mitigate | `router.use(authenticateToken)` applied before any route handler, identical to every other domain (`crm.routes.ts` line 11) — verified by acceptance criteria |
| T-22-11 | Elevation of Privilege | records.controller.ts (`:id` params referencing another company's row) | mitigate | Every handler passes `requireCompany(req)` (throws 403 if no `company_id` on the JWT) into `recordsService`, which enforces company-scoped lookups end-to-end down to the repository — no handler trusts `:id` alone |
| T-22-12 | Information Disclosure | Error responses from AppError(400/403/404) | accept | Global `errorHandler` serializes only `AppError.message`/`status`, no stack traces or internal details leak to the client (existing infrastructure, unchanged by this plan) |
</threat_model>
