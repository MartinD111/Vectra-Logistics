<objective>
Add the sub-page link block (CONT-08): inserting it via the slash menu always creates a
brand-new, empty child page (reusing `projectsApi.createPage` + `parent_page_id` nesting
verbatim — no "link an existing page" picker, per locked decision D-12) and renders a static
title+icon row that navigates to the created page on click (no live content preview, per
locked decision D-13).

This is the only content block in this phase whose `create()` factory cannot be purely
synchronous (page creation is a network call) — this plan's core design work is reconciling
that with the existing synchronous `PageBlockDef.create(): PageBlock` contract used
everywhere else, by having `create()` return an unconfigured placeholder block synchronously,
and having the block's own editor component perform the actual page-creation side effect on
mount.

Purpose: Let users organize page hierarchies inline from the block palette, matching
`PageTree.tsx`'s existing "Add sub-page" semantics exactly, with zero new backend surface.
Output: A working, insertable sub-page block; `PageCtx.pageId` threaded through so the block
knows which page is its parent.
</objective>
