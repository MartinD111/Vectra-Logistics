Output: `apps/api/src/core/config/secrets.ts` (+ test file), `server.ts` calling it before any I/O, and 4 call sites converted to lazy `getJwtSecret()` calls.
</objective>

function getEncryptionKey(): Buffer {

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Operator env/.env -> API process boot | Operator-provided config crosses into process memory at `dotenv.config()`/`bootstrap()` time — untrusted in the sense that it may be missing, empty, or a copy-pasted known-bad default |
| API process -> JWT sign/verify | `JWT_SECRET` value flows from config into every token signing/verification call across 4 call sites |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Spoofing | `apps/api/src/core/config/secrets.ts`, 4 JWT call sites | mitigate | `getJwtSecret()` rejects unset/empty/known-bad values at every call, plus `validateSecretsOrExit()` fails boot eagerly in `server.ts` before `server.listen()` — forged tokens using a guessable shared secret become impossible once no install can run with a default |
| T-14-02 | Information Disclosure | `ENCRYPTION_KEY` validation in `secrets.ts` | mitigate | `validateSecretsOrExit()` rejects unset/wrong-length/known-bad `ENCRYPTION_KEY` at boot — no install can silently run with the committed AES-256-GCM key, which would let anyone decrypt any install's stored integration secrets |

</threat_model>
