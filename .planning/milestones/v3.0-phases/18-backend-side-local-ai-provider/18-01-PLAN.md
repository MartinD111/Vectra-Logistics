<objective>
Give the backend a server-side dispatch path for the `local` AI provider so on-prem installs get full local-Gemma coverage of `aiService.complete()` and inbox/document parsing, without touching Cloud's existing hard-throw behavior for `provider==='local'`.

Purpose: this is the last unimplemented piece of `docs/specs/core/ai-integration.md` §6.1 — "Backend-only features can't use a local provider." Today `aiService.complete()` unconditionally throws for `provider==='local'` (it assumes the browser calls the local endpoint directly), which means backend-only features like inbox parsing can never use a local model even on a fully-configured on-prem install.

Output: `ai.service.ts` gains a `completeLocal` private method (mirroring `completeOpenAi`'s axios pattern, but with a 180s timeout per D-02) and a `hasUsableProvider` sibling to `hasCloudProvider` (trusts stored config only, no live ping, per D-01); `complete()` dispatches to `completeLocal` when on-prem + local + endpoint configured, and still hard-throws otherwise (Cloud unchanged). `inbox.parser.ts extract()` OR-combines `hasCloudProvider` with the new `hasUsableProvider` to unlock real extraction on local-only installs, and wraps the `aiService.complete()` call in try/catch so any local-completion failure degrades to `demoExtract` exactly like the existing non-JSON-response degrade path (D-01 corollary) — never a hard error to the dispatcher.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| API server -> company-configured `local_endpoint` | New outbound HTTP call from the backend (`ai.service.ts completeLocal`) to a URL stored in `company_ai_config.local_endpoint`. This value is company-controlled plaintext (non-secret), previously only ever called from the browser; this phase adds a server-side caller. |
| Inbox parser -> aiService.complete() -> local_endpoint | Indirect exposure of the same outbound call from `inbox.parser.ts extract()`, triggered by inbound email content (attacker-influenced trigger timing, not attacker-influenced destination). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|--------------|-----------------|
| T-18-01 | Information Disclosure (SSRF-shaped) | `ai.service.ts completeLocal()` outbound axios call to `row.local_endpoint` | accept | `local_endpoint` is set exclusively by the company's own admin via `saveConfig()` (Settings UI / installer, both already-authenticated company-scoped paths) — same trust model already accepted for the existing browser-side `completeLocal` in `ai.api.ts`. No new input surface is introduced; the value is never derived from request-time user input. On-prem admin controls their own network, matching the deployment model documented in `docs/specs/deployment/on-premise-deployment.md`. |
| T-18-02 | Denial of Service | `ai.service.ts completeLocal()` 180s axios timeout (D-02) | accept | Bounded by the explicit 180000ms timeout — a slow/unresponsive local endpoint ties up one request thread for at most 180s, not indefinitely. Matches the same bounded-timeout shape as `completeOpenAi`/`completeGemini` (60s), just wider per D-02's on-prem-hardware rationale. No new unbounded-wait surface. |
| T-18-03 | Information Disclosure | `providerError('Local model', err)` surfaces `err.message` from a failed local HTTP call | accept | Identical shape to existing `providerError('OpenAI', err)` / `providerError('Gemini', err)` calls already in production — no api key or secret exists for the local provider to leak (unlike OpenAI/Gemini, `local_endpoint`/`local_model` are stored as plaintext, non-secret columns per `ai.types.ts`). Error text is bounded to axios status/message, not raw response bodies. |
| T-18-04 | Denial of Service (degrade-path) | `inbox.parser.ts extract()` catch-and-degrade on local-completion failure | mitigate | Explicitly required by D-01's corollary — a failing/unreachable local endpoint must never surface a hard error into the inbox dispatch flow. Mitigated by design: the new try/catch degrades to `demoExtract` (deterministic, no network I/O), so a broken local AI config degrades inbox quality but never blocks or crashes the inbox pipeline. |

</threat_model>
