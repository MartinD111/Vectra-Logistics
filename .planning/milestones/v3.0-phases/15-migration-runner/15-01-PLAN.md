Log `console.log(\`Applied: ${filename}\`)` immediately after each successful commit.
    9. If the discovered-but-unapplied list is empty, log `console.log('No pending migrations.')` and exit 0 normally (no-op path for MIG-01's second-run requirement).
    10. Wrap the whole file-apply loop in a try/catch: on error, `console.error(\`FATAL: migration ${filename} failed: ${(err as Error).message}\`)` (log only `.message`, never the raw error object, to avoid leaking connection-string details from pg errors — T-15-04 mitigation) then `process.exit(1)` immediately — do not attempt any remaining files (D-04).
    11. After all migrations succeed, `console.log('Migrations complete.')`, then explicitly `await db.end();` and `process.exit(0)` so the process doesn't hang on an open pool.
    12. At file bottom: `main();` (no `await` at top level — this must remain a plain CommonJS-compatible script matching the rest of `apps/api/src`, consistent with `bootstrap()`'s invocation style in `server.ts`).

    Keep the whole file close to the ~50-line target the spec calls for — no migration library, no extra abstraction layers, just `pg` (via the imported `db`) + `fs` + `path`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| operator CLI → schema_migrations DB | `npm run migrate` is invoked by a trusted operator/CI process (via `docker compose run --rm api npm run migrate`, per D-01), not exposed to any network request path. Input surface is the local `database/migrations/` directory, not user/HTTP input. |
| migration file content → live schema | Each `.sql` file's content is executed verbatim inside a transaction with full DDL/DML privileges on the target database. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Tampering | `migrate.ts` file discovery (`fs.readdirSync`) | mitigate | Strict filename regex (`/^\d+_[\w-]+\.sql$/`) before inclusion — any non-conforming, symlinked, or path-traversal-shaped directory entry is silently skipped rather than read/executed. |
| T-15-02 | Elevation of Privilege | DB credentials (`DATABASE_URL`) used by the runner | accept | Runner reuses the same `db` Pool/credentials the API server already uses in production — no new credential surface introduced by this phase; scoping those credentials is an existing operational concern, not new to this runner. |
| T-15-03 | Denial of Service (partial state) | Multi-file migration run interrupted mid-sequence | mitigate | D-04: per-file transaction + immediate non-zero exit on first failure, no "log and continue." `schema_migrations` bookkeeping means the next invocation resumes exactly at the failed file — partial state is always resumable, never silently skipped. |
| T-15-04 | Information Disclosure | Error logging on migration failure | mitigate | Catch block logs only `err.message`, never the raw `pg` error object (which can embed connection-string/query details), before `process.exit(1)`. |
| T-15-05 | Tampering | `017_seed_admin_user.sql` accidentally applied outside dev | mitigate | D-02: hard-coded filename exclusion (`EXCLUDED_FILES` set), independent of `DEPLOYMENT_MODE` or release-packaging discipline — defense-in-depth per CONTEXT.md. |
| T-15-SC | Tampering | npm supply chain | accept | No new package installs in this phase — `migrate.ts` only imports the already-vetted, already-installed `pg` (via `../core/db`), `fs`, and `path`. No Package Legitimacy Audit required. |
</threat_model>
