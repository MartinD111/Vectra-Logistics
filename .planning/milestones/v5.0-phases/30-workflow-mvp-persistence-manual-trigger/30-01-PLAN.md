<interfaces>
The backend should define a narrow, versioned graph contract for the MVP. Exact names are Claude's discretion, but the contract should resemble:

```typescript
type WorkflowStatus = 'draft' | 'active' | 'paused';
type WorkflowRunStatus = 'queued' | 'running' | 'succeeded' | 'failed';
type WorkflowStepStatus = 'pending' | 'running' | 'succeeded' | 'failed' | 'skipped';

interface WorkflowGraphV1 {
  version: 1;
  nodes: Array<
    | { id: string; kind: 'trigger.manual'; config: Record<string, never> }
    | { id: string; kind: 'action.notification.create'; config: { title: string; body?: string; target: 'self' } }
  >;
  edges: Array<{ from: string; to: string }>;
}
```

The API surface should include the equivalent of:

```text
GET    /api/v1/workflows
POST   /api/v1/workflows
GET    /api/v1/workflows/:id
PUT    /api/v1/workflows/:id
POST   /api/v1/workflows/:id/publish
POST   /api/v1/workflows/:id/manual-runs
GET    /api/v1/workflows/:id/runs/:runId
```

If the implementation chooses slightly different verbs or paths, they must still cover list/load/save/publish/manual-run/run-detail and remain mounted through `apps/api/src/domains/index.ts`.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Frontend graph JSON -> backend workflow graph | Browser-provided nodes/config are untrusted and must be validated before persistence, publish, or run |
| RequestContext -> workflow capability checks | Build/run authorization must use the shared capability service and tenant identity from authenticated context |
| Manual trigger request -> durable run rows | Idempotency must be enforced by persisted keys so repeated clicks or retries cannot create duplicate runs |
| Workflow step execution -> notifications side effect | Step logs must record attempted side effects, successes, and failures without losing run inspectability |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-01 | Tampering | Workflow graph JSON | mitigate | Validate supported trigger/action kinds and config with DTOs before save, publish, and run; reject unsupported demo nodes server-side |
| T-30-02 | Elevation of Privilege | Workflow build/run APIs | mitigate | Assert `workflow.build` for save/publish and `workflow.run` for manual trigger from RequestContext |
| T-30-03 | Information Disclosure | Tenant-scoped workflow/run reads | mitigate | Scope every repository query by `tenant_id`; add tests that fail on cross-tenant access |
| T-30-04 | Denial of Service | Manual trigger duplicate clicks/retries | mitigate | Require/generate idempotency keys and enforce uniqueness in SQL so duplicate manual triggers reuse the original run |
| T-30-05 | Repudiation | Run/step observability | mitigate | Persist run and step logs with attempts, timestamps, errors, correlation id, and event id fields for auditability |
</threat_model>
