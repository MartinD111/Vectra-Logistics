<interfaces>
The plan should produce a typed request contract that downstream routes/controllers can consume without re-reading JWT payload fields manually. The exact file names are Claude's discretion, but the resulting surface must make these fields available in one typed object:

```typescript
interface RequestContext {
  user: { id: string; role: string; company_id: string | null; is_verified: boolean } | null;
  companyId: string | null;
  roles: string[];
  workspaceId: string | null;
  requestId: string;
  deploymentMode: 'cloud' | 'on-prem';
  deploymentCapabilities: Record<string, boolean | string>;
}
```

The capability layer must answer the Phase 28 requirement set through a shared API, for example:

```typescript
capabilities.canWorkspaceAdmin(ctx)
capabilities.canEditPage(ctx, input)
capabilities.canReadRecord(ctx, input)
capabilities.canWriteRecord(ctx, input)
capabilities.canBuildProgram(ctx, input)
capabilities.canRunWorkflow(ctx, input)
capabilities.canAdminIntegration(ctx, input)
capabilities.canAccessModule(ctx, 'crm')
```

The exact API shape may differ, but it must be reusable from later plan targets without domain-local role logic being reintroduced.
</interfaces>

| RequestContext -> capability answers | Downstream handlers will trust capability results; this layer must reflect current policy truth and deployment limits accurately |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-01 | Elevation of Privilege | Capability service | mitigate | Keep Phase 28 capability answers aligned to existing role/module/deployment truth; do not silently broaden permissions while abstracting them |
| T-28-02 | Tampering | RequestContext construction | mitigate | Derive deployment mode from validated config and actor identity from verified JWT only; do not trust downstream controllers to reconstruct those fields |
| T-28-03 | Repudiation | Shared request identity | mitigate | Include request id in the context so later event/public-edge work can correlate actions deterministically |
</threat_model>
