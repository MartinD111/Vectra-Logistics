# Milestone v5.0: Platform Foundation & Durable Execution

**Status:** Shipped 2026-07-15  
**Phases:** 27-30  
**Total Plans:** 8

## Overview

v5.0 reconciled the imported roadmap with the shipped v1-v4 platform, then hardened the request, capability, event, and workflow foundation needed before larger logistics verticals continue.

## Phases

### Phase 27: Baseline Truth & Roadmap Reconciliation

**Goal:** Maintainers can see exactly what exists, what is demo/stubbed, what still needs ADRs, and how the imported roadmap maps onto the shipped platform.  
**Depends on:** v4.0 closeout  
**Plans:** 1/1

Delivered:

- Current-state truth matrix for apps, packages, services, migrations, routes, demo/stub paths, and imported-roadmap phase status.
- Baseline command matrix for lint, typecheck, tests, builds, migrations, and local boot.
- ADR gap list for architecture-changing follow-up work.

### Phase 28: Security, Tenancy & Capabilities Foundation

**Goal:** API requests and future actions share one typed identity/capability model, and pilot domains prove tenant isolation through tests.  
**Depends on:** Phase 27  
**Plans:** 4/4

Delivered:

- Typed `RequestContext` attached by auth middleware.
- Shared capability service for workspace, page, record, program, workflow, integration, and module checks.
- Capability rollout through pilot route/domain seams.
- Trusted public request helper for gate/provider edges.
- Explicit deployment/demo fallback policy.
- Focused cross-tenant and trust regression coverage.
- Phase 28 architecture contract documentation.

### Phase 29: Event Spine & Durable Outbox

**Goal:** Meaningful service mutations can emit durable, versioned events without depending on best-effort UI calls.  
**Depends on:** Phase 28  
**Plans:** 2/2

Delivered:

- Versioned durable event envelope.
- Idempotent tenant-scoped `event_outbox` migration.
- Transactional records collection-created state+event pilot.
- Dispatcher lifecycle with bounded retry/failure state.
- Workflow-facing event catalog documentation.

### Phase 30: Workflow MVP Persistence & Manual Trigger

**Goal:** The existing automation UI stops being a mock-only surface and can save, reload, validate, manually run, and inspect a simple workflow.  
**Depends on:** Phase 29  
**Plans:** 1/1

Delivered:

- Tenant-scoped workflow draft, run, and step-log persistence.
- Authenticated `/api/v1/workflows` API.
- `workflow.build` and `workflow.run` capability enforcement.
- Manual notification-action workflow execution.
- Database-backed duplicate manual-trigger protection through idempotency keys.
- Automation dashboard and builder backed by real workflow API state.

## Milestone Summary

### Key Decisions

- Preserve v1-v4 shipped work and continue numbering at Phase 27 instead of restarting the imported roadmap.
- Make v5 foundation-first so future hierarchy, integration, automation, and logistics work reuse one request/capability/event/workflow contract.
- Keep workflow Phase 30 deliberately narrow: `trigger.manual -> action.notification.create`.
- Treat event publication as durable outbox state, while workflow run inspection lives in `workflow_runs` and `workflow_run_steps`.

### Issues Resolved

- Reconciled roadmap ambition against live repo truth.
- Added shared request/capability primitives where domains previously duplicated auth/company helpers.
- Added a durable outbox and dispatcher before workflow execution depended on events.
- Replaced automation dashboard/builder mock source of truth with persisted workflow data.

### Issues Deferred

- v4 human-UAT and `human_needed` verification items remain acknowledged deferred work.
- Full workspace hierarchy, Mini Program v3, connector lifecycle, App Store, and first logistics vertical remain future milestones.
- Broader public-token migration and production-grade workflow orchestration remain outside the Phase 30 MVP.

### Technical Debt

- Workspaces lint still has unrelated pre-existing `react/no-unescaped-entities` errors.
- Some legacy public/logistics edges remain risk areas documented by the Phase 27 truth matrix.
- Phase 28 summaries were backfilled into git at milestone close because their artifacts existed but had not been committed earlier.
