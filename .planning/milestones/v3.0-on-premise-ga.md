## Goal

Make Vectra deployable and self-upgradeable at a customer's own site as a
**first-class configuration of the same codebase** used for Cloud — not a fork,
not a "lite" build. A customer or their IT partner can install it, create their
single company + admin, run it entirely on their own infrastructure (incl. local
Gemma AI), and upgrade it later with a single documented procedure.

## Why now

CLAUDE steering §2 names On-Premise the current top strategic priority. The three
deployment specs already contain a validated, sequenced build order, and the work
splits cleanly from v2.0 (engine unification) with no overlap. Several items
(the migration runner, the committed-secret fixes, CORS/rate-limit/health
hardening) benefit **Cloud too** — this milestone de-risks both targets.

## Core value

A customer's data never leaves their server, and they can install + upgrade
without Vectra staff present and without a permanent connection to Vectra's cloud.

### Explicitly out of scope for v3.0 (tracked, not built)

- **Licensing / activation** (offline seat/module gating) — on-prem §8; real net-new work, its own later milestone.
- **Socket.IO Redis adapter** + **object storage for uploads** — cloud §3.4/§2; only needed when the API scales past one replica → a Cloud-scaling milestone.
- **Vectra-hosted AI tier** (`vectra-hosted` provider) — ai-integration §6.3; Cloud-side revenue feature.
- **Down-migrations** — release §6; backup-restore is the documented rollback, by design.

## Phase breakdown (derived from the specs' build orders)

Ordered by the on-prem §9 build order, which front-loads the fixes that also help Cloud.

### Phase 1 — Security hardening (committed secrets & seed)
**Delivers:** SEC-01, SEC-02, SEC-03
Remove the committed fallback `ENCRYPTION_KEY`/`JWT_SECRET` from every
production-facing path; make the API refuse to boot without them set. Stop
`017_seed_admin_user.sql` from running in customer-facing installs. These are
live issues independent of the on-prem timeline — do them first.
**Depends on:** nothing.

### Phase 2 — Migration runner
**Delivers:** MIG-01, MIG-02
A ~50-line runner: `schema_migrations` table, `npm run migrate` reading
`database/migrations/*.sql` in numeric order, skipping recorded files, running
the rest transactionally. Drop the initdb mounts so first-run and upgrade share
one path. Highest-leverage single piece; unblocks releases for Cloud too.
**Depends on:** Phase 1 (so the seed migration is already excluded).

### Phase 3 — Production compose + `DEPLOYMENT_MODE`
**Delivers:** DEP-01, DEP-02
`docker-compose.prod.yml` from the existing per-app production Dockerfiles (no
bind mounts, no `npm run dev`, no committed secret defaults, persistent PG/Redis
volumes). Introduce `DEPLOYMENT_MODE`, read once at boot, gating seed data and
registration (open on cloud / closed after first-run on-prem).
**Depends on:** Phase 2.

### Phase 4 — Installer / first-run flow
**Delivers:** INS-01, INS-02
Run-once installer: generate secrets (reuse `secretBox.ts`'s documented command),
create one company + real admin (replacing the fixed-credential seed), run
migrations, optionally write a local AI endpoint, write `DEPLOYMENT_MODE=on-prem`.
**Depends on:** Phases 2 & 3.

### Phase 5 — Backend-side local AI provider
**Delivers:** AIL-01
Add a `completeLocal`-equivalent in `ai.service.ts`; `aiService.complete`
dispatches to it when `provider==='local'` and the endpoint is reachable from the
server (on-prem case), replacing today's hard `throw`. Give `inbox.parser.ts` a
`hasUsableProvider` sibling so extraction gets real AI on a local-only install.
Cloud behaviour (hard `throw` for local on hosted backend) stays unchanged.
**Depends on:** Phase 3 (`DEPLOYMENT_MODE` available to gate this on-prem path).

### Phase 6 — Release versioning & upgrade docs
**Delivers:** REL-01, REL-02, REL-03
Single `VERSION` source of truth, git tag `vX.Y.Z`, stamped into images and
`/health`. Add `CHANGELOG.md` (upgrade-audience). Replace the manual `psql`
instructions in `docs/DEPLOYMENT.md` with the 5-step pull→migrate→restart
procedure. Document the additive-only / backup-restore rollback story.
**Depends on:** Phases 2 & 3.

### Phase 7 — Deploy hardening (shared with Cloud) + connectivity doc
**Delivers:** HRD-01, HRD-02, HRD-03, DOC-01
Restrict CORS + Socket.IO origins to env-configured app origins; add auth-route
rate limiting; make `/health` verify Postgres + Redis. Document the reverse-proxy
posture (expose only `/api/webhooks/*` + `/api/pod/*` publicly). Cheap, currently
silent failure modes; safe to land last.
**Depends on:** nothing hard (can parallel earlier phases); sequenced last to keep
each phase shippable.

## Sequencing summary

```
1 Security ──▶ 2 Migration runner ──▶ 3 Prod compose + DEPLOYMENT_MODE ──┬─▶ 4 Installer
                                                                          ├─▶ 5 Backend local AI
                                                                          └─▶ 6 Release + upgrade docs
7 Deploy hardening + connectivity doc  (independent; land last)
```

## Success criteria (milestone-level)

1. A fresh machine goes from `git checkout` → running Vectra with one installer
   invocation, no manual `psql`, no known-default credentials or keys.
2. `docker compose -f docker-compose.prod.yml run --rm api npm run migrate` is the
   only step needed to upgrade a customer install to a new release.
3. Every AI feature works against a local Gemma endpoint on an on-prem install.
4. `/health` reflects real dependency status and a real release version.
5. No `switch`/config path branches on being Cloud-vs-On-Prem beyond the single
   `DEPLOYMENT_MODE` axis; the same four images run in both targets.
