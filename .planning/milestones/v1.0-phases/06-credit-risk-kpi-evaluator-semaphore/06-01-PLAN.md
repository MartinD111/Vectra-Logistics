<objective>
Add a `credit_risk` KPI evaluator (RSK-01) that computes each client's credit-risk status — utilization vs. credit_limit, plus overdue-invoice payment history — as a first-class rule in the existing pluggable evaluator pattern, and wire client-subject results into `kpi_results` (already client_id-capable from Phase 1) via a new `kpi_rules.target_client_id` column. Also fix `crmService.getClientRisk()` — currently a hardcoded stub returning `{status:'unavailable'}` — to return a real computed risk status using the same logic, independent of whether a KPI rule has been run.

Purpose: This is the "first-class KPI rule" half of Phase 6. The dispatcher-facing semaphore (06-02) reads live client data directly per D-07/D-05 in CONTEXT.md — it does NOT depend on this evaluator running. This plan exists so risk is *also* queryable through the KPI engine (RSK-01's explicit requirement) and so the already-wired-but-stubbed `/crm/clients/:id/risk` endpoint and `getClientTimeline`'s kpi-result rows (`crmRepository.listClientKpiResults`) return real data.
Output: `024_kpi_target_client.sql` migration; `creditRisk.evaluator.ts` + tests; extended `kpi.types.ts`/`kpi.repository.ts`/`kpi.service.ts`/`kpiRule.dto.ts` for client-subject rules; `crmService.getClientRisk()` returns real data.
</objective>
