<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Microsoft Graph -> syncEmails() | External, potentially adversarial-content email bodies/subjects (bodyPreview, subject, sender/recipient addresses) flow into the app's DB via this method |
| BullMQ job loop -> all companies | A background job iterating every connected company must not let one company's failure (bad token, malformed Graph payload) affect any other company's sync |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Denial of Service | syncEmails() Graph pagination loop | mitigate | Loop is bounded by Graph's own `@odata.nextLink` presence (Graph paginates real data, not attacker-controlled); additionally the `$filter=sentDateTime ge {watermark}` scoping means each tick only fetches genuinely new mail, bounding page count in practice |
| T-05-05 | Tampering / Injection | Graph message fields (subject, bodyPreview, sender/recipient addresses) persisted to email_messages | accept | All values pass through parameterized `db.query` calls in `emailRepository.upsertMessages` (05-01) — no string concatenation into SQL; downstream XSS risk on render is the client-detail-page consumer's responsibility (pre-existing, out of this phase's scope) |
| T-05-06 | Denial of Service | One company's sync failure inside the BullMQ job loop | mitigate | `syncEmails()` never throws (D-10 silent-skip contract, verified by Task 1's unit tests); `Promise.allSettled` in the worker processor is defense-in-depth so even an unexpected throw in one promise cannot halt the others |
</threat_model>
