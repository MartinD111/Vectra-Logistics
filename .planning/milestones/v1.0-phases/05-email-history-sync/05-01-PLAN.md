## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Graph API -> email_messages | External Microsoft Graph payload (subject, recipients, body preview) persisted verbatim into company-scoped storage |
| email.matcher.ts -> client_id assignment | Matching logic determines which company's/client's timeline an email appears in — a matching bug could leak one client's correspondence visibility into another client's timeline |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | email_messages upsert | accept | All writes are company_id-scoped via parameterized queries; no user-supplied SQL construction — same pattern as existing calendar_events upsert |
| T-05-02 | Information Disclosure | email.matcher.ts domain matching | mitigate | Denylist (D-09) for free-mail domains prevents one client's gmail.com correspondence from domain-matching another unrelated client also using gmail.com; exact-match-only fallback for denylisted domains implemented and unit-tested in Task 1 |
| T-05-03 | Information Disclosure | email_messages read path | accept | Read path (`listClientEmails`) already exists pre-phase and already scopes by `company_id` + `client_id`; this plan does not touch it |
| T-05-SC | Tampering | npm install of `ts-node` | mitigate | `ts-node` is an already-present transitive dependency of `ts-node-dev` (existing devDependency) being promoted to a direct devDependency, not a novel package — verify version consistency against `node_modules/ts-node-dev/node_modules/ts-node`'s resolved version rather than installing an arbitrary latest |
