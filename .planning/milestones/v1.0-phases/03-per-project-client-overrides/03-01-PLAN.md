## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser -> `/api/v1/crm/clients/:id/projects[/:projectId]` | Untrusted client-supplied `project_id` (body on POST, path param on DELETE) crosses into a cross-domain write against `client_project_links`, a multi-tenant table |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege / Tampering | `crmService.upsertClientProjectLink` | mitigate | Add `assertOwnedProject(project_id, companyId)` check before `crmRepository.upsertProjectLink` — confirmed gap per RESEARCH.md Pitfall 1, closes cross-tenant IDOR on attach |
| T-03-02 | Tampering | `crmService.unlinkClientProject` (new) | mitigate | Same `assertOwnedProject` check applied before delete, plus `deleteProjectLink` repository query scopes by `company_id` in addition to `client_id`/`project_id` (defense in depth — a mismatched company_id matches zero rows even if the service check were somehow bypassed) |
| T-03-03 | Information Disclosure | `unlinkClientProjectLink` DELETE endpoint | accept | Idempotent no-op-delete convention (matches `deleteProject`/`deleteFolder`) returns 204 whether or not a row existed — does not leak whether a given `project_id` exists for another tenant, since the ownership check 404s/403s before the delete is attempted |
</threat_model>
