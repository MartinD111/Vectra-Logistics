<objective>
Build `POST /folders/tree/move` — a single, workspace.admin-gated endpoint that reparents any of folder/project/program/data_collection, validating destination tenant ownership and rejecting illegal moves (cross-tenant destinations, and — for folders only — cycles) with a specific, distinguishable error per reason. Program moves fully support both destination scopes — reparenting under a folder (`folder_id`) or under a project (`project_id`) — via the `project_id` disambiguator on `MoveNodeSchema`.

Purpose: TREEAPI-03's full delivery, closing out Phase 32. This is a thin dispatcher over already-existing, already-tested per-domain update paths — it must not reimplement cross-tenant checks or Phase 31's folder cycle detection, only orchestrate them.
Output: `foldersService.moveNode`, `POST /tree/move`, unit + live-DB coverage distinguishing cross-tenant-destination (404) from folder-cycle (400) rejections.
</objective>

From apps/api/src/domains/folders/folders.service.ts (moveFolder — the existing, Phase-31-built folder cycle/depth check, to be delegated to verbatim, never reimplemented):
```typescript
async moveFolder(ctx: RequestContext, id: string, body: unknown): Promise<Folder> {
  assertCapability(ctx, 'workspace.admin');
  const tenantId = requireCompanyId(ctx);
  const folder = await this.assertOwnedFolder(id, tenantId);
  const parsed = MoveFolderSchema.safeParse(body);
  if (!parsed.success) throw new AppError(400, parsed.error.issues[0].message);
  const { parent_id: parentId } = parsed.data;
  if (parentId) {
    const parent = await this.assertOwnedFolder(parentId, tenantId);
    if (parent.id === id || parent.ancestor_ids.includes(id)) {
      throw new AppError(400, 'Cannot move a folder into its own descendant');
    }
    // depth check, ancestor_ids rewrite, descendant patch ...
  }
  // ...
}

private async assertOwnedFolder(id: string, companyId: string): Promise<Folder> {
  const f = await foldersRepository.findFolderForCompany(id, companyId);
  if (!f) throw new AppError(404, 'Folder not found');
  return f;
}
```

From apps/api/src/domains/projects/projects.repository.ts: `updateProject(id, { folder_id })`, `updateProgram(id, { folder_id, project_id })` — existing COALESCE-update paths, return `null` if the row's `id` isn't found (repository queries are not company-scoped on the update itself, so ownership of `node_id` must be pre-checked via `findProjectForCompany`/`findProgramForCompany` before dispatch — see action notes). `updateProgram`'s patch object accepts both `folder_id` and `project_id`; passing one and explicitly clearing the other (`null`) is how a program's scope is switched between folder-filed and project-filed.

From apps/api/src/domains/folders/dto/tree.dto.ts (built in 32-01): `MoveNodeSchema = z.object({ node_type: NodeType, node_id: z.string().uuid(), new_parent_id: z.string().uuid().nullable(), project_id: z.string().uuid().nullable().optional() })`.
