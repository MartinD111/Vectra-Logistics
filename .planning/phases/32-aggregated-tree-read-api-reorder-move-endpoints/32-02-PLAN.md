<objective>
Build the lock-safe sibling-reorder primitive at the repository layer for all four node types the aggregated tree exposes (folder, project, program, data_collection): a `SELECT ... FOR UPDATE` lock on the exact sibling row set, followed by a single renumbering `UPDATE ... FROM (VALUES ...)` statement, with a `409` rejection when the client's submitted order no longer matches the live sibling set.

Purpose: This is the one genuinely new SQL technique this phase introduces (RESEARCH.md: "no fractional-position scheme, no SELECT...FOR UPDATE outside the outbox worker" existed before this phase). Isolating it at the repository layer, independent of the service/controller/route wiring (built in 32-04), lets it be tested in isolation with mocked `PoolClient`s before any transaction-orchestration or capability-gating logic is layered on top.
Output: `reorderFolders`, `reorderProjects`, `reorderPrograms`, `reorderCollections` — all `PoolClient`-scoped, lock-safe, 409-on-stale-set.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Concurrent request → shared sibling row set | Two reorder requests targeting the same parent's siblings race at the DB layer; the repository must serialize them, not the application |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-02-01 | Tampering | `reorderFolders`/`reorderProjects`/`reorderPrograms`/`reorderCollections` | mitigate | `SELECT ... FOR UPDATE` (blocking, not `SKIP LOCKED`) serializes concurrent reorders against the same sibling row set; a stale client snapshot is rejected with `AppError(409)` rather than silently applied |
| T-32-02-02 | Tampering | Renumber UPDATE statement | mitigate | `VALUES` list built with numbered `$n` placeholders and a flat params array — no string-interpolated UUIDs, even though Zod already validates UUID shape upstream (defense in depth, matches the rest of this codebase's parameterized-query convention) |
| T-32-02-SC | Tampering | npm/pip/cargo installs | accept | No new packages introduced this plan |

</threat_model>
