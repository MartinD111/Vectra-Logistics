  </read_first>
  <behavior>
    - `raceLockedTransactions(pool, txnA, txnB)` opens two `PoolClient`s via `pool.connect()` BEFORE invoking either callback (so both connections exist prior to either transaction starting, avoiding pool-exhaustion false-blocking on a small pool).
    - Records `bStartedAt = Date.now()` immediately before invoking `txnB(clientB)`.
    - Runs `txnA(clientA)` and `txnB(clientB)` concurrently via `Promise.all`, recording `Date.now()` as `aFinishedAt`/`bFinishedAt` when each resolves.
    - Releases both clients in a `finally` block regardless of success/failure.
    - Returns `{ resultA, resultB, aFinishedAt, bFinishedAt, bStartedAt }`.
    - The actual `BEGIN` / `SELECT ... FOR UPDATE` / hold-delay / `COMMIT` sequencing is the CALLER's responsibility inside `txnA`/`txnB` — this harness is a DB-agnostic orchestration wrapper only, it contains no SQL.
    - Test cases (using fake pool-like stubs — no live DB required): (1) both `connect()` calls happen before either `txnA`/`txnB` body starts executing, proven via a shared call-order array; (2) `bStartedAt` is captured before `txnB`'s callback resolves even when `txnA` is deliberately slower via a controlled `setTimeout`-wrapped delay, proving both run concurrently rather than sequentially; (3) `resultA`/`resultB` in the returned object exactly equal each callback's resolved value; (4) both fake clients' `release()` is called exactly once each, even when one callback throws.
  </behavior>
