**4. Fix `kpi_results.user_id NOT NULL`** â€” resolution chosen (Claude's discretion per CONTEXT.md, following the sketch): make `user_id` nullable and add a new `client_id` column, so a client-subject evaluator can write a result row with `user_id = NULL, client_id = <client>`, while existing user-subject rows are unaffected:
```sql
ALTER TABLE kpi_results ALTER COLUMN user_id DROP NOT NULL;
ALTER TABLE kpi_results ADD COLUMN IF NOT EXISTS client_id UUID REFERENCES clients(id) ON DELETE CASCADE;
ALTER TABLE kpi_results ADD CONSTRAINT kpi_results_subject_check
  CHECK (user_id IS NOT NULL OR client_id IS NOT NULL);
CREATE INDEX IF NOT EXISTS kpi_results_client_idx ON kpi_results (client_id, period_start DESC);
```
Wrap the `ADD CONSTRAINT` in a guard so re-running the migration doesn't error (Postgres has no `ADD CONSTRAINT IF NOT EXISTS`):
```sql
DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM pg_constraint WHERE conname = 'kpi_results_subject_check'
  ) THEN
    ALTER TABLE kpi_results ADD CONSTRAINT kpi_results_subject_check
      CHECK (user_id IS NOT NULL OR client_id IS NOT NULL);
  END IF;
END $$;
```

Use this guarded form (not a bare `ADD CONSTRAINT`) in the actual file.
