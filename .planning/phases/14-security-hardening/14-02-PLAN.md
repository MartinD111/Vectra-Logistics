<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| `docker-compose.yml` env defaults -> API container | Compose-level `:-` fallback substitution previously injected known-bad secret values into the container's environment even when the operator set nothing |
| Fresh Postgres volume -> `docker-entrypoint-initdb.d` | Any first-start Postgres volume unconditionally executed every mounted `.sql` file, including the seed-admin insert, regardless of dev vs. customer-facing intent |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-03 | Elevation of Privilege | `docker-compose.yml` postgres service `volumes:` (initdb mount) | mitigate | Remove the `017_seed_admin_user.sql` initdb mount so no fresh volume auto-creates `admin@admin.com`/`admin`; `/api/auth/signup` (existing, unmodified) is the only account-creation path left for local dev bootstrapping |
| T-14-04 | Tampering | `docker-compose.yml` `api` service environment defaults | mitigate | Strip `:-` fallback substitutions for `JWT_SECRET`/`ENCRYPTION_KEY` so an unset var reaches the container empty instead of a known-bad literal, forcing Plan 01's boot-time check to catch it rather than silently running insecurely |

</threat_model>
