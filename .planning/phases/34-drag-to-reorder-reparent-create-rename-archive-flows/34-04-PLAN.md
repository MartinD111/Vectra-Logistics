## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser client → `POST /folders`, `POST /projects`, `PATCH /folders/:id`, `PATCH /projects/:id` | Create/rename actions cross this boundary; server re-validates `workspace.admin` capability and tenant ownership on every call (unchanged this phase) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-34-06 | Elevation of Privilege | Kebab menu / right-click menu visible to any signed-in user regardless of `workspace.admin` capability (no client-side capability check exists in this codebase, RESEARCH.md Pitfall 5) | accept | Server-side `requireCapability('workspace.admin')` already gates create/rename on every endpoint this menu calls; a non-admin's action 403s and the failure is surfaced by the mutation's error path — consistent with this project's existing 403-passthrough convention (CLAUDE.md constraint) |
| T-34-07 | Tampering | Rename accepts arbitrary client-supplied `name` string | accept | Server-side Zod schemas (`UpdateFolderSchema`/project update schema) already validate the payload; this plan adds only minimal UX validation (non-empty, changed) client-side, not a duplicate authority |
