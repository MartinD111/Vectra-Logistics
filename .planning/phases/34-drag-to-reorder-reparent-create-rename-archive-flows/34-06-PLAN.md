Per D-02, dropping near a row's top/bottom edge reorders among siblings; dropping over the middle of a folder row reparents into it. Per D-04, dropping a project onto empty root-level tree space (not onto any row) un-files it — this requires its own explicit droppable region, since a row-hit-test-based `onDragMove`/`onDragEnd` has no `over` target to compute a zone from when the pointer is over empty space. Per the anti-pattern the research explicitly calls out, cycle/depth-limit legality is NEVER pre-validated client-side — the server's `AppError(400, "...")` response from `moveFolder`/`moveNode` is surfaced verbatim, and a 403 gets the friendly fallback text from the UI-SPEC copy contract.

Purpose: Deliver the two most structurally significant TREEOPS behaviors — reorder and reparent — with correct illegal-drop feedback and a working root-unfile path.
Output: `TreeNodeRow.tsx` and `TreeSection.tsx` extended with drag registration, drop-zone visuals, a root-level drop zone, and the reorder/move/un-file dispatch + error-surfacing logic.

```
interface FlatTreeRow { id: string; node_type: 'folder' | 'project'; parentKey: string; depth: number }
function flattenVisibleTree(nodes: TreeNode[], expanded: Set<string>, depth?: number, parentKey?: string): FlatTreeRow[]
function computeDropZone(pointerY: number, overRect: DOMRect, overIsFolder: boolean): 'before' | 'after' | 'into'
```

`foldersApi.reorder`/`useReorderTree()` payload: `{ node_type: 'folder' | 'project' | 'program' | 'data_collection'; parent_id: string | null; project_id?: string | null; ordered_ids: string[] }`. `foldersApi.moveNode`/`useMoveTreeNode()` payload: `{ node_type; node_id: string; new_parent_id: string | null; project_id?: string | null }`. Both endpoints scope to exactly one `node_type` per call — folders and projects sharing a parent are reordered via two separate calls, never one combined array (RESEARCH.md Pitfall 3).

New `TreeNodeRowProps` field this plan adds: `dropZone: { nodeId: string; zone: 'before' | 'after' | 'into' } | null` — must thread through the recursive `children.map(...)` call unchanged, alongside the five props 34-04 already added.

`DragMoveEvent`/`DragEndEvent` from `@dnd-kit/core` do not expose a raw pointer-Y field directly; the documented proxy for "where is the dragged element's vertical center right now" is `event.active.rect.current.translated.top + event.active.rect.current.translated.height / 2` (the active element's live translated bounding rect, not the pointer itself — `event.over.rect` gives the hovered row's own static rect for the denominator in `computeDropZone`).
