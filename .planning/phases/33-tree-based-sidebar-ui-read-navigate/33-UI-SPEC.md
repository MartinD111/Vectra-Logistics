## Spacing Scale

Declared values (must be multiples of 4), matching the scale already in active use across `WorkspaceSidebar.tsx` and sibling detail pages:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-chevron gaps within a tree row |
| sm | 8px | Row vertical padding (`py-2`), gap between stacked tree sections |
| md | 12px | Row horizontal padding (`px-3`), icon-to-label gap (`gap-3`) — matches existing `ITEMS` nav rows exactly |
| lg | 16px | Per-depth-level indent step in the tree (each nesting level adds `pl-4`); default breadcrumb segment gap |
| xl | 24px | Section padding around the new `TreeSection` heading block |
| 2xl | 32px | Vertical gap between `TreeSection` and the `ALWAYS_BOTTOM` group (existing `pt-3 mt-3` border separator pattern reused, no new value) |

Exceptions: 12px (`px-3`/`gap-3`) is a first-class token here, not just a "sm/md" rounding artifact — it is the exact value already used by every existing sidebar row in `WorkspaceSidebar.tsx:66`. New tree rows MUST reuse this value, not introduce an 8px or 16px substitute, so tree rows and fixed nav rows are visually identical in horizontal rhythm.

## Typography

Scoped to the new components this phase introduces (tree rows, section label, breadcrumbs). Existing page headings (h1/h2 `text-2xl`/`text-lg font-bold`) are unchanged and out of scope — not re-declared here.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (tree row label, breadcrumb segment) | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label (`TreeSection` heading, D-10) | 12px (`text-xs`) | 600 (semibold) | 1.2, `uppercase tracking-wider` — matches `company/[id]/page.tsx:164`'s existing section-label pattern |
| Active/current state (active tree row, current non-link breadcrumb segment) | 14px (`text-sm`) | 600 (semibold) | 1.5 |
| Display | not applicable — this phase introduces no new page-level headings; existing `text-2xl font-bold` h1 pattern is reused unchanged where breadcrumbs are inserted above it |

Declared weights (exactly 2, scoped to this phase's new UI): **400 regular** (default row/breadcrumb text) and **600 semibold** (active tree row, current breadcrumb segment, section label). Do not introduce `font-bold` (700) in new components — that weight belongs to existing page titles, not this phase's additions.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — this phase is read-only navigation; no create/write action exists (Phase 34 owns create/rename/archive CTAs) |
| Section heading (D-10, above the tree, `TreeSection`) | "Workspace" — small, non-interactive, `uppercase tracking-wider` label, matching the visual weight of the existing unlabeled `ALWAYS_BOTTOM` separator but with an explicit label since this is new organizational content (there was no tree slot before) |
| Empty state — tree has zero visible nodes after filtering (D-01/D-02 pruning removes everything) | Hide the entire `TreeSection` including its heading — do not render an empty-state message or an orphan heading with nothing beneath it. Rationale: this mirrors the existing `ITEMS.filter()` pattern, where a module with zero enabled items simply produces no row, not a placeholder. |
| Error state — `useFullTree()` query fails | Inline, non-blocking text row in place of the tree section: "Couldn't load workspace folders." in `text-sm text-gray-400 dark:text-gray-500`, no retry button this phase (React Query's automatic retry/refetch-on-focus is sufficient; a manual retry affordance is out of scope for a read-only nav feature) |
| Breadcrumbs — loading state (tree not yet resolved on a cold deep link, per RESEARCH.md's data-flow note) | Render nothing (no skeleton, no placeholder text) rather than a layout-shifting loading indicator — breadcrumbs appear once the shared tree query resolves, typically within one request round-trip since `AppShell` always mounts the sidebar first |
| Breadcrumbs — current (non-clickable) segment | Plain node name, semibold, no link styling, no trailing separator |
| Destructive confirmation | Not applicable — no destructive actions in this phase |

## Interaction Contract (phase-specific, supplements standard sections)

Because this phase is a tree/navigation UI rather than a form-heavy surface, the following interaction details are load-bearing for the checker and executor and are captured here rather than invented at implementation time:

| Element | Behavior |
|---------|----------|
| Tree row icons by `node_type` | `folder` → `Folder` (lucide), `project` → `FolderKanban` (lucide), `program` → `Zap` (lucide, matches existing `Programs`/`Automations` static nav icon), `data_collection` → `Boxes` (lucide, matches existing CRM nav icon), `project_page` → `FileText` (lucide, matches existing CMR Manager nav icon). Icon color at rest: `text-gray-400 dark:text-gray-500` (deliberately subtler than the fixed nav icons above it, to visually separate "app-level nav" from "organizational tree content"). |
| Expand/collapse control | `ChevronRight` (collapsed) / rotated 90° via `transition-transform duration-150` (expanded) — not a separate `ChevronDown` icon swap, to get a smooth rotation instead of an icon swap flash. Chevron and row label share one click target for folders (per RESEARCH.md's Open Question 2 recommendation: clicking a folder row toggles expand/collapse only, no navigation — folders have no detail page). |
| Depth indentation | Each nesting level adds `pl-4` (16px) cumulative to the row's icon+label content, independent of the row's own `px-3` base padding. |
| Non-folder row click | Navigates via the existing ID-keyed route for that `node_type` (per RESEARCH.md's `treeNodeUrl.ts` mapping) — never toggles expand/collapse (non-folder nodes have no children in the rendered sense relevant to this phase, though `project`/`program` nodes may technically have children in the raw tree; RESEARCH.md's structure implies rows for those types are still expand/collapse-capable if `children.length > 0` — apply the same chevron+click-target pattern as folders in that case, with navigation on the label and expand/collapse on the chevron only, since projects/programs DO have a real detail page unlike folders). |
| Active/current row | Same visual treatment as the existing active fixed-nav item: `bg-primary-50 dark:bg-primary-900/20 text-primary-700 dark:text-primary-400`. |
| Breadcrumb separator | `ChevronRight` at `w-3.5 h-3.5 text-gray-300 dark:text-gray-600` between segments (not a plain "/" character) — consistent with the tree's own use of `ChevronRight` for hierarchy, and consistent with lucide being the app's only icon set. |
| Breadcrumb wrapping | No ellipsis-collapse pattern (per RESEARCH.md D-07 analysis: max realistic chain ≈ 5 segments given the 3-folder-level depth cap from Phase 31). Use `flex flex-wrap items-center gap-1.5` so a long chain wraps to a second line rather than overflowing or requiring a collapse interaction. |
| Breadcrumb placement | Inserted directly above the existing page title / back-link area on the three detail views named in RESEARCH.md (`projects/[id]/page.tsx`, `projects/[id]/pages/[pageId]/page.tsx`, `collections/[collectionId]/records/[recordId]/page.tsx`) — replacing the current single-level "← Back to X" `Link` where one exists (the first two pages), and newly adding a breadcrumb row where none exists today (the record detail page, which currently has no back-navigation at all). |
