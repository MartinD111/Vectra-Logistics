<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client (browser) -> POST /api/v1/crm/clients/import | Untrusted JSON array of row objects crosses here; each field is attacker-controllable within an authenticated session |
| crm.service.ts -> crm.repository.ts / team.repository.ts | Internal boundary; company_id must always originate from the authenticated JWT (`requireCompany(req)`), never from the row payload |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | POST /clients/import body | mitigate | Reject non-array bodies with AppError(400) before iterating; every field of every row is still validated per-row via `CreateClientSchema.safeParse` (same as single-client create) — no relaxed validation path for bulk |
| T-04-02 | Information Disclosure | duplicate-VAT / employee-email lookups | mitigate | Both `findClientByVatId` and `findMemberByEmail` are scoped by `companyId` derived from the authenticated request, not from the row payload — a row cannot probe another company's clients/employees by VAT ID or email |
| T-04-03 | Denial of Service | unbounded array size in import body | accept | No explicit row-count cap added this phase (not requested); Express's default JSON body-size limit already bounds payload size. Large batches degrade gracefully (sequential per-row loop, no batch-wide transaction to roll back) |
| T-04-04 | Tampering | row-supplied `responsible_employee_id` field | mitigate | The raw row's `responsible_employee_id` (if present) is overwritten by the server-resolved id from `responsible_employee_email` before validation — a row cannot smuggle an arbitrary employee UUID from another company |
</threat_model>
