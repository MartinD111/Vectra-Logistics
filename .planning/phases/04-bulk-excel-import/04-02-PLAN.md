<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| User's local .xlsx file -> browser parse | Untrusted file content parsed client-side via `xlsx.read`; malformed/malicious spreadsheet content stays sandboxed to the browser's JS heap, never executed |
| Browser -> POST /clients/import | Parsed row objects cross to the backend as JSON; backend (Plan 04-01) is the actual trust boundary and re-validates every field — this plan does no client-side security validation, only UX-level preview |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | client-side parsed row objects before POST | accept | The frontend performs no security-relevant validation on parsed rows — all validation is enforced server-side in Plan 04-01's `CreateClientSchema.safeParse` per row; a manipulated payload sent directly to the API (bypassing the UI) is still fully validated server-side, so no client-side gap exists |
| T-04-06 | Information Disclosure | preview table rendering raw uploaded file content | accept | Preview only renders data the user themselves uploaded, in their own browser session — no cross-user data exposure risk |
| T-04-SC | Tampering | `xlsx` package usage (already an existing dependency, no new install) | accept | No new npm install in this plan — `xlsx` is already vetted/used elsewhere in the codebase (`ExcelAutomationTool.tsx`); package legitimacy gate not triggered since no new package-manager install task exists |
</threat_model>
