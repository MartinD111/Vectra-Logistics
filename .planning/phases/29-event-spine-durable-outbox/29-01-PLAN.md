<interfaces>
The durable write seam should make one domain mutation able to express an event envelope along lines such as:

```typescript
interface DurableEventEnvelope<TPayload> {
  eventId: string;
  eventName: string;
  envelopeVersion: 1;
  tenantId: string;
  actorId: string | null;
  objectType: string;
  objectId: string;
  projectId: string | null;
  causationId: string | null;
  correlationId: string | null;
  payloadVersion: 1;
  payload: TPayload;
}
```

The exact names are Claude's discretion, but the envelope must preserve the Phase 29 metadata guarantees and remain serializable into a tenant-aware outbox row.

The records pilot should be able to emit one composite event for successful collection creation, for example:

```typescript
{
  eventName: 'records.collection.created',
  objectType: 'data_collection',
  objectId: collection.id,
  payload: {
    collection: { id: collection.id, name: collection.name },
    defaultView: { id: view.id, type: view.type, name: view.name }
  }
}
```

The exact event name and payload keys may differ, but the event must stay domain-first, readable, and composite.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Service request metadata -> durable event envelope | Actor/tenant/request metadata must be normalized before the repository persists the canonical durable event |
| Repository transaction -> outbox row | The outbox row must commit or roll back with the domain mutation so no event can claim a business fact that never actually persisted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-01 | Tampering | Durable envelope contract | mitigate | Lock a versioned outer envelope with explicit tenant/object/correlation metadata so later consumers do not infer semantics from ad hoc payload shapes |
| T-29-02 | Repudiation | Pilot transaction boundary | mitigate | Persist the durable event inside the same DB transaction as the records mutation so the emitted fact always corresponds to committed state |
| T-29-03 | Information Disclosure | Multi-tenant outbox rows | mitigate | Keep the outbox first-class tenant-scoped in schema and query patterns, even for on-prem single-tenant deployments |
</threat_model>
