> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none — hand-rolled Tailwind design system already established (no `components.json` found in `apps/workspaces`) |
| Preset | not applicable |
| Component library | none (no Radix/base-ui) — custom utility classes defined in `apps/workspaces/src/app/globals.css` (`.saas-card`, `.saas-input`, `.saas-button`, `.label-xs`) |
| Icon library | `lucide-react` 0.294.0 |

**shadcn gate decision:** `components.json` was not found, but this is not a greenfield project — the codebase has 20+ shipped phases (including the directly-analogous `ClientSidebar`/`records/[clientId]/page.tsx`) built entirely on a consistent hand-rolled Tailwind system (`saas-card`, `saas-input`, `primary-*` color scale, `dark:` variants throughout). Per CLAUDE.md's explicit "Reuse over rebuild" constraint and this phase's own RESEARCH.md ("clone `records/[clientId]/page.tsx`'s structure... zero new editor code"), introducing shadcn here would fragment the design system mid-milestone for a phase whose entire job is to visually match an existing sibling page. **Decision: do not initialize shadcn for this phase.** Use the existing hand-rolled system exclusively. This is a default applied due to the non-interactive nature of this research pass — flag for user override in `23-CONTEXT.md` if they disagree.

## Spacing Scale

Declared values (must be multiples of 4), matching `records/[clientId]/page.tsx` and `DynamicBlockSettings.tsx` usage exactly:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge padding |
| sm | 8px | Compact stacked field spacing (`mb-1`, `gap-2`) |
| md | 16px | Default field/section spacing (`mb-4`, `space-y-3`/`space-y-4`) |
| lg | 24px | Panel internal padding (`saas-card` = `p-6`), section gaps (`gap-6`) |
| xl | 32px | Page horizontal padding (`px-4 lg:px-8`... capped, see exception) |
| 2xl | 48px | Page vertical padding (`py-8`) |
| 3xl | 64px | Not used at this phase's scale (reserved for future dashboard-level use) |

Exceptions:
- Property-panel row minimum touch target: **44px** height minimum for every clickable field row/button (matches the established `min-h-[48px]` convention in `records/[clientId]/page.tsx` — use 48px, not the WCAG-minimum 44px, to stay consistent with the sibling page).
- Sidebar fixed width: `w-80` (320px) — not on the 8-point scale but matches the existing `ClientSidebar` exactly; do not invent a new width.

## Typography

Matches the existing sibling page (`ClientDetailPage`) and `DynamicBlockSettings` exactly — no new sizes introduced.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label | 10px (`text-[10px]`, uppercase, `tracking-wider`) | 700 (bold) | 1.5 |
| Heading (record title) | 20px (`text-xl`) | 600 (semibold) | 1.2 |
| Display | not used in this phase — no display-scale text on the record detail page |

Note: this is 3 declared sizes (14/10/20), 2 declared weights (400/700 for label; 600 for heading — treat 600/semibold as the second weight, 700 on labels is an existing exception already baked into the sibling page's `font-bold` label convention and is carried forward unchanged rather than introducing a fourth weight).

## Color

Matches the existing `primary` (green) scale and `dark:` conventions already defined in `tailwind.config.ts` — no new colors introduced.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` / `dark: #1e293b` (`bg-white` / `dark:bg-slate-800`, via `.saas-card`) | Page background, panel/card surfaces |
| Secondary (30%) | `#f9fafb` / `dark: #0f172a` (`bg-gray-50/50` / `dark:bg-dark-bg`) | Page wrapper background, inherited (greyed) field backgrounds (`bg-gray-50 dark:bg-slate-700/50`) |
| Accent (10%) | `#16a34a` (`primary-600`) | "+ Add property" action link, active/selected property-type icon, save-in-progress indicator, focus rings on inputs |
| Destructive | `#dc2626` (`red-600`, existing `bg-red-600 hover:bg-red-700` from `UnlinkConfirmDialog`) | Delete-property action only (if included in scope), destructive confirmation button |

Accent reserved for: the "+ Add property" button/link, focus-visible ring on the active input field, the record title's edit-state indicator (if any), and the small saving/error status text next to the body editor (existing `text-red-500` pattern for errors, not accent — errors use destructive red, not accent green). Accent (`primary-600`) is NOT used for passive/decorative elements — reserve strictly for the interactive "add property" affordance and focus states, mirroring `records/[clientId]/page.tsx`'s existing usage (`text-primary-600` only appears on "Attach project"/"Override" action links).

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | "+ Add property" (button in property panel, CARD-03) |
| Empty state heading | "No properties yet" (shown only if a collection's schema is empty — edge case; primary flow always has at least a title property) |
| Empty state body | "Add a property to start tracking data on this record." + "+ Add property" button below |
| Error state (record load) | "Couldn't load this record." / "Check your connection and try again, or return to the collection." (mirrors `ClientDetailPage`'s `clientError` fallback exactly — same tone, same two-line structure) |
| Error state (property save) | "Couldn't save — try again" (inline, 11px red text under the field — exact existing pattern from `RateOverrideField`/`InlineTextField`) |
| Error state (body save) | "Couldn't save your changes — try again." (exact existing pattern from `ClientDetailPage`'s `canvasSaveError`) |
| Destructive confirmation | Not applicable this phase — CARD-01..04 include no delete-record or delete-property flow. If a "remove property" affordance is added at implementation discretion, confirmation copy must follow the existing `UnlinkConfirmDialog` pattern: "Remove this property? This deletes its value from every record in this collection. This can't be undone." |
| Untitled record placeholder | "Untitled" (shown when the title property's value is empty — matches Notion convention referenced in RESEARCH.md and `workspace-blocks.md` §4) |
| Add-property modal — name field placeholder | "Property name" |
| Add-property modal — type field label | "Type" |
| Add-property modal — submit button | "Add property" |
| Loading state | "Loading…" with spinner (exact existing pattern, `Loader2` icon + text) |

**Title source decision (blocks Pitfall 2 from RESEARCH.md):** Per RESEARCH.md's recommendation and Notion's own behavior, the record's title is sourced from the collection's **first schema property** (by array order), rendered as the `text-xl font-semibold` heading at the top of the page — not a new database column. This property should render as an inline-editable heading (reusing the `EditableTitle`-style uncontrolled contentEditable pattern from `PageHeader.tsx`, not a plain `<input>`), consistent with how every other page title in this app behaves. If the first schema property's value is empty, show "Untitled" as a placeholder (italic, greyed), matching `InlineTextField`'s existing placeholder treatment.

## Layout & Interaction Contract

(Supplemental section beyond the template's base contract — captures the composition decisions RESEARCH.md flagged as needing to be locked before planning.)

| Element | Contract |
|---------|----------|
| Route | `/collections/[collectionId]/records/[recordId]` — new path, does NOT reuse `/records/[clientId]` (route collision per RESEARCH.md Pitfall 3) |
| Page shape | Full page only (not a side-peek modal) this phase — matches `ClientDetailPage`'s exact two-column layout: `max-w-6xl mx-auto px-4 lg:px-8 py-8 flex gap-6 items-start` |
| Property panel | Left column, `w-80 flex-shrink-0`, wrapped in `.saas-card`, `dark:bg-slate-800` — exact clone of `ClientSidebar`'s shell |
| Property panel structure (top to bottom) | 1) Title (inline-editable heading, `text-xl font-semibold`), 2) schema-driven property list (each row: 10px uppercase label + type-appropriate editor, mirroring `DynamicBlockSettings`'s `label-xs` + input pattern), 3) "+ Add property" link-button at the bottom (`text-sm font-semibold text-primary-600`, same visual weight as "Attach project") |
| Record body | Right column, `flex-1 min-w-0`, `<LivePageCanvas config={record.body} onChange={setBody} />` with zero additional props (per CARD-04 and RESEARCH.md Pattern 3) — no `clientId`/`projectId` context passed since records have neither |
| Property type → editor mapping | checkbox → native `<input type="checkbox">`; date → native `<input type="date">`; select → native `<select>` populated from `property.options`; multi-select → checkbox list or multi-chip toggle (implementation discretion, but must show all selected values, not truncate silently); person → clone `EmployeeOverrideField`'s dropdown pattern (`useTeam()`); text/number/url/email/phone → plain `<input>` with matching `type` attribute for browser-native validation UX only (not the source of truth) |
| Field save trigger | Blur-commit + 800ms debounce-while-typing for text fields (exact `InlineTextField` pattern); immediate commit on change for checkbox/select/date/person (no debounce needed — discrete value change) |
| Body save trigger | 1500ms debounce after edit, seed-once on first load, flush-on-unmount (exact `ClientDetailPage`/`projects/[id]/pages/[pageId]` pattern) — no new autosave mechanism |
| Add-property flow | Two sequential requests, never combined (RESEARCH.md Pitfall 1): (1) `PATCH /collections/:id` with updated `schema` array, awaited; (2) only on success, `PATCH /records/:id` setting the new prop's initial value. Modal/inline form collects: property name (required, text input), property type (required, select of the 12 supported types), and if type is `select`/`multi-select`, an inline "add option" list builder writing `{id, label}[]` into `options` |
| Cross-tenant / not-found | 404 (record not found or wrong company) renders the exact same two-line error shell as `ClientDetailPage`'s `clientError` block — never assume any URL-supplied ID is valid |
| Touch targets | All interactive rows/buttons in the property panel: `min-h-[48px]` (matches sibling page exactly, not a new convention) |
