<objective>
Make `GET /health` perform a live per-request check of Postgres and Redis reachability instead of returning a static `{status:"OK"}`, per D-01/D-02/D-03 (HRD-03).

Purpose: closes the "health check doesn't verify dependencies" gap in `docs/specs/deployment/cloud-deployment.md` §3.3 — load balancers/orchestrators can now detect a genuinely unhealthy instance rather than keeping a silently-broken one in rotation.
Output: a new `apps/api/src/core/health/health.service.ts` exporting a pure, DI-friendly `checkDependencyHealth()`, wired into `server.ts`'s `/health` handler, with the existing `server.health.test.ts` updated to tolerate the new (unconnected-in-test-process) dependency behavior per RESEARCH.md Pitfall 1.

Runs in wave 2, after 20-01, because both plans edit `apps/api/src/server.ts` (20-01 touches the CORS/Socket.IO block, this plan touches the `/health` handler — same file, sequenced to avoid conflicting concurrent edits).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Load balancer / orchestrator (unauthenticated) → `GET /health` | No auth required by design (health probes must be reachable pre-auth); response reveals dependency reachability state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-08 | Denial of Service (silent unhealthy instance kept in LB rotation) | `GET /health` | mitigate | Live `Promise.allSettled` dependency probe via `checkDependencyHealth()`, no caching per D-02 (Task 2) |
| T-20-09 | Information Disclosure (dependency topology exposed to unauthenticated caller) | `GET /health` response body | accept | Response reveals only `postgres`/`redis` reachability state (`'ok'`/`'down'`), never connection strings, credentials, or error messages — standard practice for health endpoints, no PII/secrets exposed |
| T-20-10 | Denial of Service (health-check-triggered DB/Redis load under high polling frequency) | `db.query`/`redisClient.ping` on every request, no caching (D-02) | accept | `SELECT 1`/`PING` are sub-millisecond on a healthy connection; health probes are typically low-frequency (5-30s intervals) per D-02's explicit rationale |
</threat_model>
