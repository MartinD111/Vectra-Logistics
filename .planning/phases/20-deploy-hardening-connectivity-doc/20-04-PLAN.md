    1. One or two sentences of framing: if placing Vectra behind a reverse proxy or firewall, only two route prefixes need to be reachable from outside the network.
    2. A markdown table with columns `Route | Why it must be public | Notes`, exactly two rows:
       - `/api/webhooks/*` — Samsara and Geotab push GPS/telematics events to the server via signed, HMAC-verified webhook; there is no polling alternative, so if this route isn't reachable, telematics updates stop. Notes: only needed if the customer uses Samsara or Geotab integration.
       - `/api/pod/*` — drivers upload signed proof-of-delivery photos from a single-use link on their phone, often from the road, not the office network. Notes: deliberately unauthenticated (token-scoped per link), by design.
    3. A paragraph stating everything else — the three frontend apps, the rest of the API, admin tooling — should stay on the internal network/VPN; there is no functional requirement for the general API or UI to be internet-reachable.
    4. An "Outlook / Microsoft 365 setup" callout: if live Outlook integration is enabled, the OAuth callback needs to be reachable by whichever browser completes the admin's sign-in at setup time — this does not require public internet if the admin configuring it is on the LAN/VPN, but will fail if the admin is external with no ingress at all. Cross-reference the existing `## Outlook / Microsoft 365 integration` section further down in the same file for the exact callback URL (`MS_REDIRECT_URI` / `/api/v1/outlook/callback`) rather than restating it.
    5. A "Recommended posture" paragraph: run a reverse proxy (nginx, Caddy, Traefik, or a cloud load balancer) that exposes only `/api/webhooks/*` and `/api/pod/*` publicly, routing everything else so it's reachable only from the internal network or VPN. Note that a fully air-gapped install (no public ingress at all) will lose telematics webhooks and public POD upload links — a real, conscious tradeoff, not a bug.
