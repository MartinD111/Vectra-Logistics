<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Docker host network -> postgres/redis | Only `api`/`matching-engine` containers should reach these; no host-port exposure in prod |
| Operator's `.env`/secrets store -> compose interpolation | Missing required secrets must fail the compose invocation itself, before any container starts |
| Public internet -> marketplace/workspaces/cmr/api host ports | These are the only services that should be reachable; Postgres/Redis must never be |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-16-04 | Information Disclosure / Elevation of Privilege | postgres/redis host-port exposure | mitigate | D-03: no `ports:` key on either service in docker-compose.prod.yml — reachable only via internal compose network; admin access via `docker compose exec`, not an open host port (V14 Configuration) |
| T-16-05 | Tampering | Missing/soft-defaulted secrets at the compose layer | mitigate | `${VAR:?message}` required-variable syntax on `POSTGRES_PASSWORD`, `JWT_SECRET`, `ENCRYPTION_KEY`, `DEPLOYMENT_MODE` — compose itself refuses to start any container if any is unset, complementing (not duplicating) the app-level `validateSecretsOrExit()`/`validateDeploymentModeOrExit()` fail-fast from 16-01-PLAN.md (V14 Configuration) |
| T-16-SC | Tampering | Base images (`postgres:15-alpine`, `redis:7-alpine`, `node:18-alpine`, `python:3.11-slim`) | accept | All 4 base images are official Docker Hub images already pinned and running unmodified in the existing dev `docker-compose.yml`/Dockerfiles — no new image sources introduced by this phase, no npm/pip/cargo package installs added by this plan (compose YAML + one .env.example line only) |

</threat_model>
