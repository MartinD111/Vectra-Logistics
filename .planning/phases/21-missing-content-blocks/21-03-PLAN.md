<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| User-pasted URL string → `<img src>`/`<a href>`/`<video src>`/`<iframe src>` | The URL is stored and rendered as-is (no server-side fetch); the browser, not this app's server, resolves the URL when rendering `<img>`/`<video>`/`<iframe>` |
| Table cell contentEditable text → `.textContent` only (no innerHTML) | Same trust boundary as Plan 21-01's checklist/quote — plain text extraction only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|--------------|-------------------|
| T-21-07 | Tampering / SSRF | Bookmark/embed "link preview" | mitigate | Explicitly implemented WITHOUT a server-side fetch (D-06 cheapest-default path) — hostname/URL string is parsed and rendered entirely client-side via `new URL()`, eliminating the SSRF surface a scraping endpoint would introduce. Verified by the Task 3 acceptance criterion that no `fetch(` call exists in `MediaBlocks.tsx`. |
| T-21-08 | Tampering | `<iframe src>` built from user-pasted YouTube/Vimeo URL | mitigate | The embed `src` is reconstructed from an extracted numeric/alphanumeric video ID matched against a strict allowlisted hostname set (`youtube.com`, `youtu.be`, `vimeo.com` and `www.` variants only) — arbitrary hostnames fall through to the plain `<video>` tag, never an `<iframe>`, so a malicious URL cannot cause an arbitrary-origin iframe to be embedded |
| T-21-09 | Tampering | `<a href={block.url}>` / `<img src={block.url}>` with arbitrary user-pasted URL | accept | Standard client-side link/media rendering already used everywhere else in the app (e.g. document links); browsers already sandbox cross-origin resource loads; `rel="noreferrer"` is applied on all outbound anchor links to avoid leaking `Referer` |
| T-21-10 | Denial of Service | Oversized/malformed media URLs | accept | No file upload path is introduced in this plan (URL-only, D-05) — there is no server-side storage or processing of attacker-supplied bytes to exhaust |
</threat_model>
