<objective>
Add the two nesting content blocks this phase introduces — toggle (CONT-02) and multi-column
layout (CONT-07) — sharing one `children: PageBlock[]`-shaped mechanism per D-01's discretion
(a fixed 2-lane `ColumnsBlock.columns: PageBlock[][]` reuses the exact same nested-rendering
component as `ToggleBlock.children`, satisfying "one mechanism... simple to render/edit").
Nesting is capped at one level (D-02) and only content-group kinds may nest (D-03), enforced
both at the data layer (the nested insert menu is filtered to `nestable: true` items, which
never includes `toggle`/`columns` themselves) and the UI layer.

This is genuinely new ground — no block has ever had children before this phase (confirmed by
21-RESEARCH.md across the whole codebase, including mini-program blocks). Per RESEARCH.md's
Anti-Patterns and Pitfall 3, this plan deliberately does NOT reuse `LivePageCanvas.tsx`'s
`@dnd-kit`-driven full grid+DnD approach inside the nested children list (a second
`SortableContext` nested inside the first risks drop-target collisions) — instead it builds a
purpose-specific, simpler `NestedBlockList` component: an ordered list with an "+ Add block"
insert-only affordance (reusing `InsertBlockMenu`, extended with an optional pre-filtered
`items` prop) and optional up/down move buttons — no drag-and-drop inside nested children.

Purpose: Ship both nesting requirements with the cheapest mechanism that stays simple to
render/edit, exactly matching the user's stated preference (CONTEXT.md's "Specific Ideas"
section: "default to the lower-cost implementation when in doubt").
Output: Toggle and multi-column blocks, insertable, collapsible/laid-out, with nested-children
add/remove/reorder-via-buttons, both correctly capped at one level of nesting.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Nested child block editing → same `PageConfig` JSONB persistence path as top-level blocks | No new persistence boundary — nested children are just more deeply-nested `PageBlock` objects inside the same JSONB column |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|--------------|-------------------|
| T-21-13 | Tampering | A malicious/corrupted `PageConfig` JSON containing a `toggle`/`columns` block whose `children`/`columns` array itself contains another `toggle`/`columns` block (bypassing the UI-level D-02 cap, e.g. via direct API manipulation) | mitigate | `NestedBlockList`'s `renderChild` dispatch goes through `pageBlockRegistry.renderEditor`, which will render whatever kind is present (including a rogue nested toggle) — but since the nested toggle/columns would ALSO try to render ITS OWN `NestedBlockList` recursively, this is bounded (not unbounded recursion, since a real attacker would need to hand-craft deeply nested JSON via direct API calls, not the UI) and each level still only reads/writes the same company-scoped, auth-gated `project_pages.config` column. No stack-overflow-inducing unbounded depth is possible from normal UI use; deliberately not adding server-side JSON depth validation in this plan since it is a frontend-only phase — flagged here for awareness, not a blocking gap |
| T-21-14 | Tampering | Nested child contentEditable text (checklist/quote/table cells rendered inside a toggle/column) | accept | These blocks were already hardened in Plans 21-01/21-03 (`.textContent`-only commits, no `innerHTML` for these kinds) — nesting them does not introduce a new sanitization surface |
</threat_model>
