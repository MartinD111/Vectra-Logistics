## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser contentEditable HTML (including mention spans) → `DOMPurify.sanitize()` → `PageConfig` JSONB → API PATCH | User-authored HTML crosses into persisted storage; this is the existing rich-text/list trust boundary, extended (not replaced) by this plan |
| Mention click → same-app/new-tab navigation | `@page` mentions build a URL from a company-scoped `pageId` already resolved server-side via `projectsApi.listPages` (auth-gated) — no user-supplied raw URL is used for navigation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|--------------|-------------------|
| T-21-04 | Tampering | Mention span HTML (`data-mention-type`/`data-mention-id` attributes) | mitigate | `DOMPurify.sanitize(html, { ADD_ATTR: [...] })` at BOTH sanitize call-sites (edit-commit in `EditableRichText.tsx` and read-render in `pageBlockViews.tsx`) — an explicit, narrow allowlist extension, not a blanket attribute allow; only these two `data-*` keys are added, everything else stays on DOMPurify's default (safe) allowlist |
| T-21-05 | Tampering | `data-mention-id` value spoofing (a user could hand-edit the DOM to point a mention span at an arbitrary id before it's re-sanitized) | accept | The mention id is only ever used to build a company-scoped API lookup (`teamApi`/`projectsApi`, both already `authenticateToken`+`company_id`-scoped) — a spoofed id at worst 404s or resolves to a different in-company record the user already has read access to via those endpoints; no privilege escalation possible |
| T-21-06 | Information Disclosure | `@person` hover card showing name/email | accept | Same company-scoped `teamApi.list()` data already visible on the `/team` page to any authenticated user in the company — no new exposure |
| T-21-SC | Tampering | No new npm/pip/cargo packages installed by this plan | accept | Package Legitimacy Gate not applicable — zero new dependencies |
