# Phase 12 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase's only UI surface is one new page block, `Callout`, added purely via a `WorkspaceBlockPlugin` registry entry (EXT-01/EXT-02). There is no new page, layout, or navigation. This contract intentionally reuses 100% of existing project-page chrome, spacing, color tokens, and the slash-menu UI — it defines only what the callout block itself needs: its visual variant and its inline-edit interaction. No new design tokens, layout systems, or interaction patterns are introduced.

## Component Contract (Callout block)

Since this phase's entire scope is one block, the component-level contract is the design deliverable:

**Read view (`CalloutView`):**
- Container: `flex items-start gap-2 rounded-xl border px-3 py-2.5 text-sm` + fixed info-blue tone classes above
- Icon: `MessagesSquare` from `lucide-react`, `w-4 h-4 mt-0.5 shrink-0` — **must** use an icon name already present in `components/projectPage/icon.tsx`'s `MAP` (verified present). Do not add a new icon or edit `icon.tsx` (would break EXT-01 criterion 3's file-touch scope).
- Text: plain string, no HTML rendering (`{block.text || 'Empty callout'}`) — no `dangerouslySetInnerHTML`, no DOMPurify needed (V5 input validation: plain text only, zero XSS surface, per research).

**Edit view (`CalloutEditor`):**
- Same container/icon as read view, for visual continuity between read and edit modes (no jarring layout shift when entering edit mode).
- Inline edit pattern: **always-editable contentEditable div** (not click-to-edit-then-commit) — matches the existing pattern used by `EditableHeading.tsx` and every other text-bearing block on the canvas. This is the established project convention; do not invent a click-to-edit toggle for this one block.
- Uncontrolled `contentEditable` with the mandatory focus guard (`if (!el || document.activeElement === el) return;`) copied verbatim from `EditableHeading.tsx` to avoid caret-jump bugs (research Pitfall 3).
- Commit on blur only (`onBlur` reads `textContent`, trims, calls `onUpdate` if changed) — matches existing block editors; no autosave-per-keystroke, no Enter-to-submit special-casing needed (single-line-or-wrap text, unlike heading's Enter-to-blur behavior — not required here, but harmless if copied).

**Both views share identical container markup/classes** — the only difference is `contentEditable` + `onBlur` on the edit view's text node. This keeps the "read → edit" transition visually seamless, matching the rest of the block library.
