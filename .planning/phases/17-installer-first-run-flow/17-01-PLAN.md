function getEncryptionKey(): Buffer {
  const hex = process.env.ENCRYPTION_KEY;
  if (!hex || hex.length !== 64) {
    throw new AppError(500, 'ENCRYPTION_KEY env var is missing or invalid (must be 64-char hex)');
  }
  return Buffer.from(hex, 'hex');
}

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Operator CLI input → install.ts | Company name, admin email, admin password entered interactively (readline) or via flags/env vars by the customer's IT partner — a trusted-but-fallible human, not an anonymous network caller |
| install.ts → PostgreSQL | Direct `pg` writes with full DB privileges (ops tool, not a scoped app role) |
| install.ts → filesystem (.env) | Writes secrets in plaintext to `.env`, consumed by `docker-compose.prod.yml` at next boot |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-17-01 | Spoofing / Info Disclosure | `generateSecrets()` | mitigate | `crypto.randomBytes(32).toString('hex')` CSPRNG for both `JWT_SECRET`/`ENCRYPTION_KEY` — never a fallback/placeholder; unit-tested for 64-hex-char format and per-call uniqueness (Task 1) |
| T-17-02 | Spoofing | `createCompanyAndAdmin()` | mitigate | Real customer-provided email/password only, `bcrypt.hash(password, 10)` — no default/known credential path; `017_seed_admin_user.sql` already hard-excluded from the runner (Phase 15 defense-in-depth) |
| T-17-03 | Information Disclosure | `main()` prompt/flag handling | mitigate | Admin password accepted only via `INSTALL_ADMIN_PASSWORD` env var or interactive prompt — never a `--admin-password` CLI flag (avoids shell history/`ps` exposure); confirmation echo never includes the password; failure logging uses `(err as Error).message` only, never the raw error object |
| T-17-04 | Tampering | `createCompanyAndAdmin()` SQL | mitigate | Parameterized queries (`$1, $2, ...`) throughout — no string concatenation of company name/email into SQL |
| T-17-05 | Tampering / Repudiation | `applyBaseSchema()` | mitigate | `to_regclass('public.companies')` guard prevents re-running non-idempotent `CREATE TYPE` statements against an already-initialized database, which would otherwise throw or (if ignored) leave the DB in an inconsistent state |
| T-17-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this phase — `pg`, `bcrypt`, `crypto`, `fs`, `path`, `child_process`, `readline/promises` are all pre-existing dependencies or Node built-ins already in `apps/api/package.json`/runtime |
</threat_model>
