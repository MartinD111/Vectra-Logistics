## Spacing Scale

Declared values (must be multiples of 4), matching existing usage in `CrmClientsBlock.tsx` / `PageTree.tsx`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps (`gap-1`), badge inline padding |
| sm | 8px | Compact row/element spacing (`gap-2`, `space-y-2`), table cell vertical rhythm |
| md | 16px | Default card padding (`p-4`), section-to-section gaps within a panel |
| lg | 24px | Sidebar-panel to main-canvas gutter, dashboard page horizontal padding |
| xl | 32px | Layout gaps between dashboard header and table |
| 2xl | 48px | Major break between dashboard toolbar and empty-state illustration block |
| 3xl | 64px | Page-level top/bottom margin on the client detail page |

Exceptions: none. Table cell padding uses `px-4 py-4` (16px vertical, the `md` token) — grid-aligned; the existing `(fleet)/management/page.tsx` table uses `py-3.5` (14px), but matching a non-grid-aligned table isn't sufficient reason to break the scale, so this phase's tables use `py-4` instead. Click-to-edit inline fields on the detail-page sidebar (D-10) use a minimum 48px touch/click target height regardless of text size — grid-aligned (matches the `2xl` token) and still exceeds the 44px accessibility minimum for touch targets.

## Typography

Matches existing scale used in `CrmClientsBlock.tsx` (`text-sm`, `text-xs`, `text-[10px]`) — extended with a heading size for the dashboard/detail page titles. Declaring exactly 4 sizes, 2 weights:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label | 12px (`text-xs`) | 600 (semibold) | 1.4 — used for table headers, field labels, badges (uppercase tracking-wide per existing `.label-xs` for the smallest 10px variant, reserved for micro-badges only) |
| Heading | 20px (`text-xl`) | 600 (semibold) | 1.2 — client name on detail page, dashboard page title "CRM" |
| Display | 28px (`text-3xl` scaled down from typical 30px to fit the 8pt-adjacent scale) | 600 (semibold) | 1.2 — reserved for the top-level "CRM Dashboard" page heading only |

Weights: 400 (regular, body text/table data) and 600 (semibold, headings/labels/emphasis). No 500 or 700 usage introduced this phase — matches existing `font-semibold`/`font-bold` mixed usage in `CrmClientsBlock.tsx`; standardize new code on 600 (`font-semibold`) only, avoid `font-bold` (700) to keep exactly 2 weights.

## Color

Existing `primary` scale in `tailwind.config.ts` is a green scale (`primary-500 = #22c55e`, `primary-600 = #16a34a`) — this is the app's accent, NOT the credit-risk semaphore color (which uses raw `red`/`amber`/`emerald` Tailwind colors per `CreditBar` in `CrmClientsBlock.tsx`, unrelated to Phase 6's frosted-glass block warning). This phase reuses both without introducing new hues.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-white` / `dark:bg-slate-800` (page/panel background) | CRM dashboard page background, client detail page canvas background |
| Secondary (30%) | `bg-gray-50`/`bg-gray-100` (light) / `dark:bg-slate-800`/`dark:bg-slate-700` (dark) — `saas-card` surfaces, sidebar panel | Dashboard table container, detail-page sidebar settings panel, table row hover state |
| Accent (10%) | `primary-600` (`#16a34a`) / `primary-700` hover | Reserved for: "Add client" button, "CRM" sidebar nav active state/icon, primary CTA buttons ("Add client", "Save" affordance if ever surfaced, "New client page" menu entry), active/focused input ring |
| Destructive | `red-500`/`red-600` (Tailwind default red, matches existing `CreditBar` over-limit color) | Reserved for: over-limit credit badge/row indicator on the dashboard table, "over limit" warning text, any future delete-client confirmation (not in this phase's scope but color reserved now for consistency) |

Accent reserved for: the "Add client" button, "CRM" sidebar active nav highlight, primary form submit actions, and focus rings on interactive inputs. Never used for body text, card borders, or informational badges — those stay gray/slate per the dominant/secondary split above.

Credit-status semaphore colors (dashboard column, D-02) reuse the exact 3-tier logic from `CreditBar` in `CrmClientsBlock.tsx`: `bg-emerald-500` (under 70% utilization), `bg-amber-500` (70–99%), `bg-red-500` (at/over limit) — do not introduce new thresholds or colors for this phase's dashboard badge.

## Layout & Component Contracts (phase-specific, beyond template defaults)

### CRM Dashboard (`/records`)
- Full-page data table (D-01) — NOT the compact card style of `CrmClientsBlock`. Container: page padding `p-6` (24px, `lg` token), table wrapped in a `saas-card`-style surface (white/slate-800, rounded-2xl border) OR a flatter bordered table matching `(fleet)/management` table markup (`w-full border-collapse`, `thead`/`tbody`, `px-4 py-4` cells — 16px vertical, grid-aligned) — use the flatter table pattern since this is a data-dense list, not a card grid.
- Columns (D-02, fixed order): Name (semibold, links via row click) → Country (badge, same style as `CrmClientsBlock`'s `text-[10px] font-bold px-1.5 py-0.5 rounded bg-gray-100` country tag) → Credit status (utilization bar/badge reusing `CreditBar` visual logic, condensed to fit a table cell — badge form: colored dot + percentage, not the full bar, to keep row height compact) → Responsible employee (name text, or "Unassigned" in `text-gray-400 italic` if null).
- Row click → opens client detail page in new tab via `crossAppUrl`/`target="_blank" rel="noopener noreferrer"` pattern from `packages/ui/src/appUrls.ts` (NAV-02). Row shows `cursor-pointer` and `hover:bg-gray-50 dark:hover:bg-slate-800/60` affordance.
- Toolbar (D-03): search input (`saas-input`/`input-field` style, left-aligned, ~280px width) + "Over limit only" toggle (pill/switch, right of search) + "Add client" button (right-aligned, primary accent). All three sit in one horizontal row, `gap-3`, above the table, `mb-4`.
- "Add client" (D-04): modal dialog, reuses the exact form fields/validation from `CrmClientsBlock.tsx`'s inline form (name, country select, VAT ID, credit limit, default rate) — same field order, same `saas-input` styling, same default values (country defaults `'DE'`, limit defaults `'10000'`). Modal shell: centered overlay, `saas-card` surface, `max-w-md`, close via backdrop click or explicit X icon (`lucide-react` `X`).

### Client Detail Page (`client_pages`)
- Layout (D-09): two-column — persistent left sidebar (fixed width ~320px, `saas-card` or bordered panel, `dark:bg-slate-800`) for editable settings fields; main area is the block canvas (reuses `LivePageCanvas.tsx`/`PageBlockView.tsx` rendering contract) holding "current situation" and "timeline" blocks stacked full-width.
- Sidebar fields (address, notes, responsible employee) use inline click-to-edit (D-10): default render as static text with a subtle hover affordance (faint border/background on hover, no visible "Edit" button); click transforms into an input/textarea in place with a minimum 48px click/touch target height (grid-aligned, exceeds the 44px accessibility minimum); autosave on blur or debounce (~800ms) while typing; show a small inline spinner or checkmark transiently on save, matching the low-chrome Notion-like feel already used in `EditableRichText.tsx`/`LivePageCanvas.tsx`.
- Responsible-employee field (D-13): dropdown/combobox listing all team members (reuses existing team/users list source), flat list, no grouping — render selected value as plain text with a chevron affordance on hover, consistent with other sidebar fields.
- "Current situation" section (D-11): heading + last-10-emails list. Empty state (no `email_messages` yet): centered icon (`Mail` from `lucide-react`, `text-gray-300`) + "No emails synced yet" heading + body copy from Copywriting Contract. Do not hide the section — always render the heading and container.
- Timeline (D-12): single reverse-chronological feed, each entry showing a small icon badge by type (email = `Mail`, invoice/billing = `Receipt` or `FileText`, chart/KPI event = `BarChart3`, all from `lucide-react` to match existing icon sourcing), timestamp in `text-xs text-gray-400`, and a one-line description. No tabs/filters. Empty state: "No activity yet" per Copywriting Contract, same visual treatment as the current-situation empty state (icon + heading + body, centered, `py-8` vertical padding matching `CrmClientsBlock`'s existing empty-state pattern).
- New tab creation entry point (D-07/D-08): "New client page" item added to the existing project page creator's add-page menu (exact menu component to be located during planning — likely `SlashMenu.tsx` or the page-creation trigger in `PageHeader.tsx`/`PageTree.tsx`). Selecting it opens a small picker modal (search existing clients + "Create new client" quick-create fallback), then either opens the existing `client_pages` row in a new tab (if one exists, D-08) or creates one and opens it — never creates a duplicate.
