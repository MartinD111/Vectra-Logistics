## Color

No new hues introduced. Reuses the exact palette established in Phase 2 (`tailwind.config.ts` `primary` green scale as accent; slate/gray as dominant/secondary; red as destructive/semaphore).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-white` / `dark:bg-slate-800` | Linked-project row/card background |
| Secondary (30%) | `bg-gray-50`/`bg-gray-100` (light) / `dark:bg-slate-700`/`dark:bg-slate-800` (dark) | Section container background, inherited-value placeholder background (greyed-out field state per D-04) |
| Accent (10%) | `primary-600` (`#16a34a`) / `primary-700` hover | Reserved for: "Attach project" trigger button, the active/entered state of an overridden field's border or icon (signals "this is user-set"), focus ring on the project picker search input |
| Destructive | `red-500`/`red-600` | Reserved for: the "Unlink project" button/icon only. Not used for the per-field "clear override" action — clearing an override is a low-consequence, reversible action (see Copywriting Contract) and must NOT look destructive; use neutral gray for that icon/button instead. |

Accent reserved for: "Attach project" button, overridden-field visual indicator (icon color or left border accent on the field when `is_overridden.<field>` is true), focus rings on the picker input. Never used for the unlink button (that's destructive/red) or for inherited/greyed-out field text (that stays gray per Secondary role).

**D-04 override-state color contract (explicit, since this is the phase's core visual differentiator):**
- **Inherited (not overridden):** field value rendered in `text-gray-400 italic` on a `bg-gray-50 dark:bg-slate-700/50` placeholder background — visually identical treatment to Phase 2's "Unassigned"/empty-placeholder styling, signaling "this is not this project's own value."
- **Overridden:** field value rendered in `text-gray-900 dark:text-white` (full-strength text, not italic) on the normal `bg-white dark:bg-slate-800` surface, with a small `primary-600` left border accent (`border-l-2 border-primary-600`) or a filled `primary-600` dot/icon next to the label — the single unambiguous visual signal required by D-04.

## Layout & Component Contracts (phase-specific, beyond template defaults)

### "Linked Projects" Section — Placement & Structure
- **Placement (Claude's Discretion, resolved):** Added as a new section in the main canvas column (to the right of `ClientSidebar`), stacked above the existing "current situation"/timeline blocks — NOT inside the 320px-wide `ClientSidebar` itself. Rationale: the sidebar is a fixed-width settings panel for single-value fields; linked-project rows need horizontal room for 3 override fields + actions per row, which the 320px sidebar cannot comfortably fit. Section wrapped in a `saas-card` matching the canvas block visual style, full width of the main column.
- **Layout treatment (Claude's Discretion, resolved):** List of cards, one per linked project — not a dense table. Each card can expand/collapse (default: collapsed, showing project name + a compact summary of which fields are overridden, e.g., small colored dots/badges for rate/employee/notes). Clicking the row expands to show all 3 fields with their full override affordances. This matches the low-chrome, progressive-disclosure feel of the rest of the detail page (matches `LivePageCanvas` block conventions) and avoids a wall of always-expanded override controls when a client has many linked projects.
- Section heading "Linked Projects" styled as a `label-xs`-weight section title (10px uppercase semibold, matching other section labels), with the "Attach project" button right-aligned on the same row.

### Linked Project Card (collapsed state)
- Row: project name (14px, `font-semibold`, `text-gray-900 dark:text-white`) + compact override indicator (3 small dots or a "2 overrides" text badge, `primary-600` for overridden count) + chevron (expand affordance) + "Unlink" icon button (right-aligned, `red-500`, 48px click target).
- Hover state: `hover:bg-gray-50 dark:hover:bg-slate-800/60`, matching existing row hover conventions from the CRM dashboard table.

### Linked Project Card (expanded state)
- Three field rows: Rate, Responsible employee, Notes — each following the D-04 override-state color contract above.
- Each field row layout: label (10px uppercase) → value display (greyed/italic if inherited, full-strength + left-border accent if overridden) → action button ("Override" if inherited, "Reset to default" if overridden) right-aligned on the same row or directly below on mobile-width breakpoints.
- Rate field: numeric input when in override-editing state (reuses `saas-input` styling, `type="number"`, EUR suffix/prefix label consistent with any existing rate display in `CrmClientsBlock`).
- Responsible employee field: same dropdown/combobox pattern as `ResponsibleEmployeeField` from Phase 2 (flat team list, no grouping), but must show a third "Use default" option distinct from "Unassigned" — since inherited vs. explicitly-overridden-to-null-employee are different states. Overriding to "Unassigned" (explicitly no one, project-specific) must remain visually distinguishable from "using the global default" (which itself might resolve to Unassigned).
- Notes field: same `InlineTextField` multiline pattern as Phase 2, with the override toggle wrapping around it (toggle switches between "show greyed global notes" and "editable override textarea").

### Attach Project Picker (Claude's Discretion, resolved)
- **Form factor:** Dropdown-style popover anchored to the "Attach project" button (not a full modal) — consistent with the low-friction feel of `ResponsibleEmployeeField`'s existing dropdown pattern (`fixed inset-0` backdrop + absolutely positioned panel). A full modal is unnecessary overhead for a single search+select action.
- Content: search input at top (`saas-input`, `Search` icon prefix, placeholder "Search projects…"), scrollable filtered list below (`useProjects()` client-side filtered, per RESEARCH.md's confirmed no-server-search-needed guidance), each row showing project name; clicking a row immediately attaches (no separate "confirm" step — matches the low-friction "upsert as idempotent attach" backend behavior).
- Already-linked projects are visually deprioritized in the list (e.g., shown with a checkmark and `text-gray-400`, not hidden) so the user understands re-selecting is safe (idempotent) rather than confusing/erroring.

### Interaction Pattern for Per-Field Override Toggle (Claude's Discretion, resolved per D-04)
- **Chosen widget:** Explicit text button labeled "Override" / "Reset to default" (not a pencil icon, not click-to-edit-directly). Rationale: D-04 requires the override-vs-inherited state to be "visually unambiguous" — an icon-only affordance (pencil) risks being confused with Phase 2's always-editable inline-field pattern, which this phase must deliberately NOT resemble. An explicit labeled button makes the state transition (inherited → overridden, and back) a conscious, visible action rather than an implicit "click to edit" gesture reused from a different interaction contract.
- Clicking "Override" transitions the field into an editable state pre-filled with the current global value (a sensible starting point for the user to then adjust) and immediately begins tracking it as an override once changed/saved — matches Pitfall 2's guidance that the mutation must always send the full current override state (all three fields), not just the one being edited.
- Clicking "Reset to default" immediately clears that field's override (single API call sending `null` for that field while preserving the other two fields' current override values, per Pitfall 2) and reverts display to the greyed/inherited state — no confirmation dialog (per Copywriting Contract).
