D-02 prop-type validator design (from 22-RESEARCH.md Pattern 3 / 22-PATTERNS.md, first-party — no existing analog):
```typescript
type PropType = 'text' | 'number' | 'date' | 'select' | 'multi-select'
  | 'checkbox' | 'person' | 'url' | 'email' | 'phone' | 'files' | 'relation';

function validatePropValue(type: PropType, value: unknown): boolean {
  switch (type) {
    case 'text': case 'url': case 'email': case 'phone': case 'select':
      return typeof value === 'string';
    case 'number': return typeof value === 'number';
    case 'checkbox': return typeof value === 'boolean';
    case 'date': return typeof value === 'string';
    case 'multi-select': case 'files': case 'relation':
      return Array.isArray(value) && value.every((v) => typeof v === 'string');
    case 'person': return typeof value === 'string';
    default: return true;
  }
}
```
Service-level `validateProps(schema: CollectionPropertyDef[], props: Record<string, unknown>): void` loops every key in `props`, finds the matching schema entry by `id`, throws `AppError(400, 'Unknown property: ' + key)` if not found, throws `AppError(400, ...property name... + ' must be a valid ' + type)` if `validatePropValue` returns false.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| controller (untrusted req.body) → service | Service is the sole place where DTO envelope validation (Zod) and D-02 prop-type validation happen — no write reaches the repository unvalidated |
| service → repository (cross-entity trust) | Service must re-fetch the parent collection scoped to `companyId` before trusting its `schema` for prop validation — never trusts a `collection_id` without an ownership check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-07 | Tampering | createRecord / updateRecord (props vs. schema) | mitigate | D-02 validateProps rejects any prop value whose type doesn't match the collection's declared schema, with AppError(400) before any repository write — verified by Task 1 Tests 2-4 |
| T-22-08 | Elevation of Privilege | createRecord (collection_id references another company's collection) | mitigate | `getCollection(collection_id, companyId)` is always called with the caller's `companyId` before trusting the collection's schema — a cross-tenant `collection_id` resolves to `null` → AppError(404), never leaks another company's schema |
| T-22-09 | Tampering | createView/updateView (`config` JSONB, opaque per REC-03) | accept | `config` is intentionally unvalidated beyond DTO envelope (`z.record(z.string(), z.unknown())`) — filters/sorts/groupBy semantics are frontend-owned in later phases (23-26); low risk since config is never executed server-side, only stored/retrieved |
</threat_model>
