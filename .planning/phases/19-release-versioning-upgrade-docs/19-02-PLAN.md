<objective>
Stamp the root `VERSION` value into all 4 production Docker images via build `ARG`, avoiding the `apps/api` vs frontend build-context asymmetry (RESEARCH.md Pitfall 1), per REL-01.

Purpose: A built image should carry the version it was built from, independent of which of the 4 build contexts (`./apps/api` vs repo root) produced it.
Output: `ARG VERSION` in all 4 Dockerfiles, an `args:` block with `VERSION` in `docker-compose.prod.yml` for all 4 app services.
</objective>

  <action>
    Edit `apps/api/Dockerfile` (single-stage, 13 lines): insert `ARG VERSION=unknown` and `ENV VERSION=$VERSION` on two new lines immediately after the `FROM node:18-alpine` line (line 1) and before `WORKDIR /app`. This is the same `apps/api` image that `version.ts` reads `process.env.VERSION` from at runtime (Plan 01), so `ENV` (not just `ARG`) is required here — `ARG` alone does not persist into the running container.

    Edit `apps/marketplace/Dockerfile`, `apps/workspaces/Dockerfile`, `apps/cmr/Dockerfile` (each multi-stage, identical shape): insert `ARG VERSION=unknown` on a new line immediately after the `FROM base AS runner` line, and `LABEL org.opencontainers.image.version=$VERSION` on the next line, before `WORKDIR /app`. `ARG` scope does not cross `FROM` boundaries in multi-stage builds, so it must be redeclared in the `runner` stage specifically (not just added near the top `FROM node:18-alpine AS base` line) — the frontends only need image metadata (`LABEL`), not a runtime `ENV`, since they don't read `process.env.VERSION` at request time.
  </action>
