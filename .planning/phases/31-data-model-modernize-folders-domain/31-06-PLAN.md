    In a `before` hook: insert two `companies` rows (distinct `id`s via `gen_random_uuid()` or explicit UUIDs), insert a folder `A` in company 1 and a folder `B` in company 2 directly via `db.query('INSERT INTO folders (company_id, name) VALUES ($1, $2) RETURNING *', ...)`.

    Test 1 (HIER-02, DB level): attempt `UPDATE folders SET parent_id = $1 WHERE id = $2` setting folder B's (company 2) `parent_id` to folder A's id (company 1) directly via `db.query` — assert this rejects. Since the composite FK requires `(parent_id, company_id) REFERENCES folders (id, company_id)` and folder A's `(id, company_id)` pair doesn't match folder B's `company_id`, this must throw a Postgres foreign-key-violation error (`error.code === '23503'`) — assert via `await assert.rejects(() => db.query(...), (err) => err.code === '23503')`.

    Test 2 (HIER-03, DB level): create folder `C` as a child of folder `A` (same company), then attempt `UPDATE folders SET parent_id = <C's id> WHERE id = <A's id>` (moving A into its own child C) directly via `db.query` — assert this rejects with the trigger's raised exception (`err.message` includes "descendant", `err.code === 'P0001'` or the `check_violation` SQLSTATE `'23514'` depending on how `RAISE EXCEPTION ... USING ERRCODE` maps — assert on `err.message` containing "descendant" as the primary check, since that's stable regardless of exact SQLSTATE mapping).

    Test 3 (HIER-03, depth): create a 3-level chain (A → C → D, all same company, depths 1/2/3), then attempt to create a 4th-level folder as a child of D directly via `db.query('INSERT INTO folders (company_id, parent_id, name) VALUES ($1, $2, $3)', ...)` — assert this rejects with a message containing "depth".

    In an `after` hook: delete all fixture rows created (folders first, then companies), using `db.query('DELETE FROM folders WHERE company_id = ANY($1::uuid[])', [testCompanyIds])` then `db.query('DELETE FROM companies WHERE id = ANY($1::uuid[])', [testCompanyIds])`, and call nothing else (do not `db.end()` — the pool is shared with any other test file that might run in the same process).
