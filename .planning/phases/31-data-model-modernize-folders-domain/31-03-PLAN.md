<objective>
Replace the existing hard `DELETE /projects/:id` and `DELETE /projects/programs/:id` routes with archive/unarchive endpoints (D-01, D-02 as corrected by RESEARCH.md: this is an edit of existing hard-delete code, not net-new removal functionality), and add the bulk, transaction-safe (`PoolClient`-taking) repository methods the folders-domain cascade (plan 31-05) needs to archive projects/programs/pages when a folder is archived, per HIER-04 and RESEARCH.md's Pitfall 5 two-pass cascade (folder-filed projects/programs in pass 1, then that project's own programs/pages attached via `project_id` in pass 2).

Purpose: Per RESEARCH.md's Pitfall 3, this plan intentionally does NOT migrate `projects.service.ts` onto the v5 `RequestContext`/`event_outbox` pattern wholesale (that modernization is scoped to "the folders domain" per HIER-05/06's wording) — it only replaces the delete methods with archive/unarchive and adds cascade-support bulk methods, keeping `recordEvent()` for all other unrelated project/program mutations (create/update) unchanged.
Output: Projects and programs support `archived_at` with working archive/unarchive endpoints; `projectsRepository` exposes bulk cascade-archive primitives.
</objective>
