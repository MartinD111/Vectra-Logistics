<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client → `POST /folders/:id/archive`, `/unarchive` | Net-new mutation endpoints — must carry the same `workspace.admin` capability gate as every other folder mutation |
| `folders.service.ts` cascade transaction → `projects`/`records` repositories | Cross-domain repository calls inside one transaction; a partial failure must roll back all tables, not just `folders` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-03 | Elevation of Privilege | `POST /:id/archive`, `/:id/unarchive` | mitigate | Both routes wired with `requireCapability('workspace.admin')` in `folders.routes.ts`, identical to every other folder mutation route — Task 3 acceptance_criteria explicitly checks this |
| T-31-01 | Tampering / Elevation of Privilege | `moveFolder`'s cross-tenant `parent_id` acceptance | mitigate | Service-layer `assertOwnedFolder` (404 for cross-tenant) is defense-in-depth on top of the DB-level composite FK from plan 31-01 |
| T-31-02 | Denial of Service | `archiveFolder`'s cascade transaction across up to 6 row types | mitigate | Bounded by max depth 3 (D-04) and GIN-indexed `ancestor_ids` lookups — no unbounded recursion; the transaction rolls back atomically on any failure so a partial cascade never persists |
| T-31-04 | Tampering (partial cascade / lost event) | Cascade archive event emission | mitigate | All `insertDurableEvent` calls happen inside the same `BEGIN`/`COMMIT` as the row updates — if the transaction rolls back, no orphaned events are written; `event_outbox`'s `ON CONFLICT (tenant_id, event_id) DO NOTHING` prevents duplicate events on retry |
| T-31-SC | Tampering (supply chain) | N/A — no new packages | accept | No new npm dependency introduced by this plan |
</threat_model>
