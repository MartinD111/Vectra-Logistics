## Gaps Closed During This Audit

**HIER-03 / HIER-07 — Folder move was not atomic and did not re-validate descendant depth.**

Originally found by Phase 31's own verification (`31-REVIEW.md` CR-01/CR-02), left unresolved through phase 31 sign-off. Re-surfaced by this milestone audit's integration checker, which confirmed the vulnerable code path (`folders.service.ts moveFolder`) is directly reachable via the Phase 34 drag-to-reparent UI shipped this milestone — i.e., a real user performing the feature this milestone exists to deliver could trigger silent data corruption (a descendant folder silently exceeding the depth-3 invariant) or a stale ancestor-index on a failed cascade.

Fixed in commit `9747730`:
- Descendant depth is now computed relative to the moved folder and validated before any write; the move is rejected with 400 if any descendant would exceed `MAX_FOLDER_DEPTH`.
- The folder's own row update and the descendant `ancestor_ids` patch now commit in a single transaction (matching the existing `archiveFolder` pattern), so a mid-cascade failure can no longer leave descendants with stale ancestor_ids.
- 3 regression tests added (48/48 folders-domain tests pass); `npx tsc --noEmit` clean across both `apps/api` and `apps/workspaces`.

Given the fix was scoped, well-understood, low-risk (mirrors an existing pattern in the same file), and directly protects the feature this milestone shipped, it was applied inline rather than deferred to a separate closure phase.
