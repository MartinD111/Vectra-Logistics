# fleet.md — Fleet module (vehicles, drivers, telematics, spot quotes, exceptions)

Scope: the Fleet business module — `apps/api/src/domains/fleet/` and its
frontend blocks. This is a **vertical module**, not core platform (per
`CLAUDE.md` §1: named vertical blocks/domains are an accepted product layer,
built on top of the generic primitives, not folded into them). This documents
what's real, what's a labelled demo, and — the headline finding — a live
telematics pipeline that receives real data and then throws it away.

> Suggested location: `docs/specs/modules/fleet.md`.
> Reads with: `event-spine.md` (fleet writes to it today — nowhere, see §5),
> `on-premise-deployment.md` §7 (Samsara/Geotab webhooks are the inbound
> dependency documented there).

---

## 1. What exists — module map

`apps/api/src/domains/fleet/`:
- `fleet.controller.ts` / `fleet.service.ts` / `fleet.repository.ts` /
  `fleet.types.ts` — CRUD for **drivers** and **vehicles**.
- `dispatcher.controller.ts` — the three "Phase 2 dispatcher widget" endpoints:
  telematics snapshot, spot quote calculator, exception radar.
- `telematics.service.ts` — live fleet positions + AETR driving-hour status.
- `spotQuote.service.ts` — break-even/suggested rate calculator.
- `exceptions.service.ts` — operational crisis feed (border delays, port
  congestion, wagon damage, engine faults).

Frontend: `components/projectPage/FleetTelematicsBlock.tsx`,
`SpotQuoteBlock.tsx`, `ExceptionRadarBlock.tsx` — registered `PageBlockKind`s
(`fleet-telematics`, `spot-quote`, `exception-radar`) per `workspace-blocks.md`.

Schema: `vehicles`/`drivers` in `init.sql` (base schema, pre-numbered
migrations); `fleet_exceptions` in `014_dispatcher_widgets.sql`.

## 2. Drivers & Vehicles CRUD — real, complete, but thin

`fleet.service.ts`/`fleet.repository.ts` is straightforward, working
`company_id`-scoped CRUD. Nothing mocked here. Current fields:

```
Driver: first_name, last_name, phone, email, license_number, status
Vehicle: license_plate, vehicle_type, max_weight_kg, max_volume_m3, max_pallets
```

This is thinner than the business plan's Fleet module description (§7.7:
"vozil, voznikov, prikolic, servisov in vzdrževanja, dokumentacije,
GPS/telematičnih povezav"). Concretely missing, in priority order:
- **Trailers** — no table/entity at all; `vehicle_type` is a free string, not
  a tractor/trailer distinction.
- **Maintenance/service records + reminders** — no schema for service history
  or due-date tracking.
- **Document expiry** (insurance, technical inspection/roadworthiness) — a
  generic `company_documents` table exists but isn't vehicle-scoped or
  reminder-driven.
- **Driver compliance data beyond a license number** — AETR hour totals are
  computed live in `telematics.service.ts` (see §3) but never attached to the
  driver record itself; there's no historical AETR ledger, only the current
  live snapshot.

None of these need new platform primitives — they're straightforward new
columns/tables in this vertical domain, same shape as what exists.

## 3. Telematics — the headline finding: a complete pipeline that discards its own data

This deserves to be read carefully; it's the most consequential gap in the
module and easy to miss because every individual piece looks reasonable in
isolation.

### 3.1 The webhook receiver is real
`on-premise-deployment.md` §7 already documents this: `POST /api/webhooks/
samsara` and `/geotab` are properly HMAC-verified (`verifySamsaraSignature`
does constant-time comparison against `SAMSARA_WEBHOOK_SECRET`; Geotab checks
a bearer token the same way) and correctly refuse to boot without a secret in
production. **The security layer is production-grade.**

### 3.2 The payload processing is a stub
`webhook.service.ts::processSamsaraWebhook` / `processGeotabWebhook` parse the
incoming GPS payload correctly (Zod-validated against `SamsaraWebhookSchema`/
`GeotabWebhookSchema`), then **only `console.log` it**. Both functions contain
an explicit TODO: *"Cross-domain interaction — update Fleet vehicle location…
Requires: mapping Samsara vehicle ID → our internal vehicle ID via a
`samsara_vehicle_map` table"* — that mapping table doesn't exist, and there is
**no `updateVehicleLocation` method, no vehicle-positions table anywhere.**
Every real GPS event a customer's telematics provider sends today is received,
verified, logged, and discarded.

### 3.3 The read path never reads real data either
`telematicsService.getSnapshot` checks `integration_credentials` for a
`connected` Samsara/Geotab row (`connectedProvider()`), and **when found,
still calls `buildDemoFleet()`** — synthetic positions generated by a seeded
PRNG (`mulberry32`/`hashString`), just tagged with the real provider's name so
the UI looks connected. The code comment is honest about this: *"Real adapter
goes here… Until credentials-backed calls are wired, serve synthetic
positions."* AETR hour math (`aetrFor`, correctly implementing the EC
561-2006 continuous/daily/weekly limits) is real logic applied to fake input.

**Net effect**: a customer can connect Samsara, their trucks can be pushing
real GPS to Vectra's webhook right now, and the dispatcher's "My Fleet" board
will still show a deterministic fake truck looping a hardcoded European
corridor. The frontend does label this honestly (`data.demo` → "showing demo
telematics" badge) — the dishonesty is only in the backend's silent
"connected but still fake" branch, which the demo badge doesn't distinguish
from the true not-connected case in the same way the code comments do.

### 3.4 What closing this actually requires
Three pieces, in dependency order:
1. **A vehicle-positions store.** A `vehicle_positions` table (or, more in
   the spirit of `event-spine.md`, treat each GPS ping as an `activity_event`
   with verb `fleet.position.updated` and the lat/lng in `payload` — this
   gets timeline/history "for free" from the spine's existing query patterns,
   and avoids a bespoke table for what's fundamentally a stream of timestamped
   facts about a vehicle). Prefer the event-spine route unless the position
   stream volume argues for a dedicated hot-path table — worth a deliberate
   choice, not a default.
2. **A provider-ID mapping** (`samsara_vehicle_map(company_id,
   samsara_vehicle_id, vehicle_id)`, same shape for Geotab) so an inbound
   webhook's provider-specific ID resolves to Vectra's `vehicles.id`.
3. **Wire `processSamsaraWebhook`/`processGeotabWebhook`** to actually write
   through that mapping into whichever store §1 chooses, and change
   `telematicsService.getSnapshot` to read real positions when a provider is
   connected, falling back to `buildDemoFleet()` **only** when genuinely
   disconnected (i.e. delete the "connected but still fake" branch — that's
   the actual bug to fix, not just an enhancement).

Until this is done, be precise in any product/customer-facing description:
telematics **integration status** and **webhook ingestion** are real; **live
position display** is not, regardless of connection state.

## 4. Spot Quote calculator — real, self-contained, worth reusing as a pattern

`spotQuote.service.ts` is a genuine deterministic cost model, not a mock:
great-circle distance × a road-factor constant → toll (`TOLL_EUR_PER_KM` per
country) + fuel (`FUEL_EUR_PER_L` × `CONSUMPTION_L_PER_100KM` per equipment
type) + driver-day cost + overhead → break-even and margin-adjusted suggested
rate. Every intermediate figure is returned so a dispatcher can sanity-check
it — a good transparency pattern worth reusing anywhere else the platform
does a computed estimate (e.g. a future ETA or cost feature).

Data note: the city list, toll rates, and fuel prices are **hardcoded arrays/
maps in this file**, not tenant-configurable and not sourced from a live feed.
This is acceptable as vertical-module reference data (not a platform-core
violation — it lives entirely inside `domains/fleet`), but it will drift
(diesel prices move month to month) and the city list is European-only and
fixed. If this becomes a real pricing tool rather than a rough estimator,
externalise the rate tables (company-editable, or a periodic feed) rather than
editing source code to update a fuel price — same principle as the Program
Builder reference tools' "user-editable lookup tables" pattern in
`program-builder.md` §0.

## 5. Exception Radar — real, persisted, correctly realtime, but not on the event spine

`exceptions.service.ts` is solid: `fleet_exceptions` is a real table,
`create()` persists and pushes `exception:new` over the realtime bus to the
company's room, `simulate()` provides a demo generator for demoing the live
push without a real integration connected — a good pattern (compare to
`ai-integration.md` §4's "always ship a working fallback," here applied to a
realtime feature rather than an AI one).

**Gap**: nothing in `apps/api/src/domains/fleet/` calls `recordEvent` —
confirmed by searching the whole domain. Per `event-spine.md` §3's hard rule
("write from the SERVICE layer… every meaningful action"), this domain is
currently invisible to statistics, KPIs, and activity timelines. Concretely
missing verbs (naming per `event-spine.md` §4's `domain.action` convention):
- `driver.created` / `driver.updated` / `driver.removed`
- `vehicle.created` / `vehicle.updated` / `vehicle.removed`
- `fleet.exception.created` / `fleet.exception.resolved` (this one especially —
  exception frequency/resolution time is an obvious KPI candidate once
  `kpi-engine.md`'s `on_time_delivery`-style evaluators exist)

This is a cheap, mechanical fix — no schema change, just add the same
`recordEvent(...)` call pattern already used correctly in `projects.service.ts`
/ `team.service.ts` to each fleet mutation.

## 6. Do / Don't

**Do**
- Describe telematics precisely to anyone (internally or customer-facing):
  webhook security and ingestion are real; live positions are simulated
  regardless of connection status, until §3.4 ships.
- Add `recordEvent` calls to every Fleet CRUD and exception mutation — this
  domain is currently a blind spot for the KPI/statistics layer.
- Treat the Spot Quote breakdown transparency (§4) as the pattern for any
  future computed-estimate feature.
- Choose deliberately between an `activity_events`-based position stream and a
  dedicated table (§3.4) rather than defaulting to a new table out of habit.

**Don't**
- Don't present "provider connected" as equivalent to "live tracking working"
  until the webhook→storage→read chain in §3.4 is actually wired end to end.
- Don't treat the hardcoded toll/fuel/city tables in Spot Quote as a platform
  violation — they're fine as vertical-module reference data, just flag them
  as needing an update mechanism before relying on them for real pricing.
- Don't add a second event-logging mechanism for Fleet — use `recordEvent`
  exactly as every other domain does.
